## モチベーション
- **完遂感**: PR がスムーズにマージされ、難しい実装も含めて気持ちよく終わること
- **自律性**: 周囲に邪魔されず、自分のペースで決められる感覚が最大の報酬
- **直接的な貢献の感謝**: 同僚から「翔太のおかげで助かった」と伝わること
- **オフの充実**: 定時に仕事を終え、夜は趣味・娯楽に安心して集中できること
- **迷いのなさ**: デザイナーの意図がクリアで、迷いなく実装に入れること；ツールが揃って作業できること
- **軽い同期への許容**: カジュアルに「観てもらう／観に行く」程度の同期参加ならば受け入れられる

### 補足 / 留意点
- 「同僚からの感謝」（Q3 B）と「自律性（邪魔されない）」（Q2 C）は両立する価値観として保持。どちらかを優先しなければならない局面では、自律性が優先される可能性が高い（Q2 で自律性を選択したことから）。

## 障害
- **受信側の割り込み**: 急な Slack メンションが集中を断ち切る（集中を奪う一番の原因）
- **半集中状態の疲弊**: 返事待ち中も「来たかな」と気になり、作業が完全に再開できない
- **計画の侵食**: 他チームのペースに引きずられて、自分の作業計画が崩れる
- **情報の散在**: Slack・Notion・GitHub・Jira が散らばっており、情報を探す時間が発生する
- **組織の将来不安**: 今のチーム・会社にずっといてよいのかという漠然とした不安
- **個人では動けない**: 課題はチームや組織の構造にあると認識しており、一人では改善できないという無力感

### 補足 / 留意点
- モブが嫌いなのではなく「自分に無関係な場が嫌い」という可能性が高い。逆に言えば、自分に関係するテーマのカジュアルな場なら参加できる余地がある。
- **T1 保留トレードオフの整合**: 「Slack に投げる（非同期・自分発信）」はデフォルト行動だが、「Slack でメンションされる（受信・割り込み）」が最大の障害。「非同期 vs 同期」という単純な軸ではなく、「自律的なタイミングで繋がれるかどうか」が翔太さんにとっての本質的な問いだと想定される。

## アウトカム
- **集中時間の確保 → 挑戦できる**: 深く集中できる時間が増えて、技術的にチャレンジングな実装にも臆さず挑めるようになる
- **邪魔のない仕事の流れ**: 自分のペースで進められる仕事の流れが確立されており、誰にも余計に邪魔されない状態
- **待ちからの解放**: 「誰かを待つ」ことがなくなり、自分のスケジュールで仕事を完結できる
- **充実した積み重ね**: キャリアの節目に「あの時期は充実していた」と思えるような積み重ねができている
- **ここにいる意味の言語化**: 「なぜここで働いているか」の納得感・理由が持てている
- **オフの完全な回復**: 夜や週末は仕事を引きずらず、ゲーム・映画・読書に心おきなく集中できる
- **チームの変容**: チームや他のメンバーも使うようになり、コミュニケーション品質がチーム全体で上がる（個人ツールではなく、チームで広がることで真価を発揮すると想定）

## 検証ポイント（インタビューで確かめるべき不確実な点）

1. **「カジュアルな同期」の定義**: 「観てもらう／観に行く」が受け入れられると回答したが、具体的にどんな形式・時間・頻度なら実際に参加するか。「カジュアル」の解釈は個人差が大きく、プロダクト設計の核心に関わる。
2. **組織不安の深度**: 「会社の将来が不安」が単なる不満レベルなのか、転職を具体的に検討しているのか。リテンション効果を設計要件に入れるかどうかを左右する。
3. **定時志向 vs 技術どっぷり の優先度**: 「やりたい技術にどっぷり浸かれる」（ざっくり Q7）と「定時で終わらせたい」（モチベーション Q4）の実際の優先度。業務外でも技術投資をするタイプか否か。
4. **Slack 送受信の非対称性**: 自分が投げる（送信）は普通にするが、受け取る（割り込み）が最大の障害というパターン。ターゲット層で同様のパターンが再現するか確認が必要。これはプロダクトの通知設計の根幹になる。
5. **チームへの広がりが必要条件か**: 「チームや他のメンバーも使うことで価値を感じる」（アウトカム Q7）と回答したが、翔太さん一人だけが使うプロダクトでは採用しないのか、「広がれば最高」程度のニュアンスか。導入戦略（個人→チーム vs チーム一括導入）を左右する。
