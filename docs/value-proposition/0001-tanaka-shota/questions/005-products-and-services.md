> Synthesis で採用された 5 つのマッチ案（M1: 集中している側を弱者扱いしない場 / M2: 気配を流す UX / M3: 集中文化コミュニティ / M4: 声を出さない MTG / M5: 中抜きアプローチ）を、具体的な提供物の形態に落とし込みます。

## Q1. 主たる提供物の形態は次のうち、どれが最もイメージに近いですか? （複数選択可）

- A. SaaS（個人サブスクリプション）
- B. デスクトップアプリ（Mac/Windows のネイティブ）
- C. ブラウザ拡張（Slack 等の既存サービス上に介入）
- D. コミュニティ／プラットフォーム
- E. 書籍・コンテンツ・講座
- F. 上記の複合（ハイブリッド）
- G. 自由記入:

回答: F（A + C + D の複合）
> 主軸は SaaS（A）。Slack 等への介入は拡張（C）。長期的にコミュニティ（D）に広げる。

---

## Q2. 主機能セットは次のうちどれを核にしますか? （Synthesis の採用マッチからの落とし込み・複数選択可）

- A. M1 由来: 送信側に「集中している相手」を弱者扱いしない規範を伝える UI（メンション送信時の介入）
- B. M2 由来: 言葉にしない意図（あとで返す／緊急じゃない／気配だけ伝えたい）を相互伝達する UX
- C. M5 由来: 集中時間中は情報を "なくす" 設計（受信側で意図的に届かせない）と、終了時にダイジェストが届く仕組み
- D. M4 由来: テキスト主体・非同期主体の MTG（「声を出さない MTG」モード）
- E. M3 由来: 集中文化を育てるコミュニティプラットフォーム
- F. 自由記入:

回答: A, B, C（MVP）／ D, E（V2 以降）
> MVP は A + B + C を核にする。D は次フェーズでの統合機能。E は長期構想。

---

## Q3. 利用時のタッチポイントは?

- A. デスクトップアプリ（Mac/Windows）— OS の DnD・通知制御と直結
- B. ブラウザ拡張 — Slack Web 上で送信側 UI に介入
- C. Slack アプリ — ステータス連動・通知バンドル
- D. モバイルアプリ — 通知一括管理
- E. 自由記入:

回答: A, B, C
> A: 受信側体験（情報を "なくす"）／ B: 送信側規範形成（M1 の核）／ C: チームへの自然な広がり

---

## Q4. 単発か継続かの形態は?

- A. 月額サブスクリプション
- B. 買い切り
- C. フリーミアム（個人無料・チーム有料）
- D. プロジェクト課金
- E. 自由記入:

回答: C
> 「個人の集中体験を起点に、チームに広がる」M3 の長期視点と整合する設計。
