# Synthesis — エージェント主導の統合

ステップ 3「エージェント主導の統合」の詳細手順。**このスキルの核**。両側（ペルソナの抽出済み Customer Profile と、提供者側の棚卸し）が独立して揃ってから、エージェントが分析の重労働を担う。

> **発散と検証を分ける原則**: マッチ案の発散はエージェントの責任範囲。「新しい案を思いつくため」の選択肢を質問シートに列挙してユーザーに委ねると、選択肢自体がバイアスとなり思考を狭める。したがって、シート 006 に載せるのは**エージェントがすでに提示したマッチ案 M1, M2, ... に対する採用/修正/不採用の検証設問**だけ。分析（A / B）と提案（C）はエージェントで済ませ、ユーザーには判断だけを渡す。

## ステップ全体像

```
A. ペルソナの根本原因分析   ┐
                           ├→ C. 非自明なマッチポイントの発見 → ユーザー検証
B. 提供者の本質分析         ┘
```

A と B は順番を問わず並行的に行ってよい。重要なのは **C で両者を突き合わせる前に、A と B をそれぞれ独立して深く掘る**こと。Aを「Bと結びつきやすい角度」で歪めない、Bを「Aに刺さりそうな角度」で歪めない。

---

## A. ペルソナの根本原因分析（Root Cause Analysis）

ステップ 1 で抽出した Pains を表層のまま受け取らない。**「なぜこのペルソナはこの状況に陥っているのか」をメカニズム・構造・因果まで掘る**。

### 掘り方

各主要 Pain について次を問う:

1. **直接の原因**: その Pain が発生する直前の引き金（イベント・刺激）は何か?
2. **構造的原因**: その引き金が発生する**仕組み**は何か?（誰が・何が・どんなルールで・どんな前提で動いているか）
3. **メタ原因**: その仕組みが存在する**背景**は何か?（文化・制度・歴史・技術的制約）
4. **相互作用**: 複数の Pain がどう連鎖しているか? P1 が P2 を呼び、P2 が P3 を悪化させる…のような連鎖は?

### Pain と Gain の関係

Gains を「Pain の裏返し」だけにしない。

- **Pain の解消**（−1 → 0）に対応する Gain
- **積み増し**（0 → +1、+1 → +5）に対応する Gain
- それぞれメカニズムが異なるので、両者を区別して掘る。

### アウトプット

各 Pain / 主要 Gain について **1 〜 3 行のメカニズム言語化**。

例:

> P1（Slack 割り込みで集中が断ち切られる）
> - 直接の原因: Slack のメンション通知が集中時間中に届く
> - 構造的原因: 即時応答が文化的に強要されており、応答しないことに罪悪感や評価低下のリスクが付随する
> - メタ原因: チャットツール普及期に「即時応答 = 礼儀」とする文化が定着した
> - 連鎖: P1 が P2（半集中の疲弊）を生み、P2 が継続することで P3（計画侵食）が常態化する
> - **本質的な軸**: 「自律的なタイミングで繋がれるかどうか」が翔太の本質的な問い。「同期 vs 非同期」より深い軸。

## B. 提供者の本質分析（Essence of Provider）

ステップ 2 で集めた棚卸し情報を、**表層と深部に切り分ける**。

### 切り分け

- **表層**: 既存商品・サービス・現在の事業ドメインなど、目に見える形で組織が提供しているもの。
- **深部**: マインドセット・ビジョン・ノウハウの背骨・組織が本質的に持つケイパビリティ。**他領域に転用できる**性質を持つ。

例:
- 表層: 「集中タイマーアプリ」「タスクダッシュボード」
- 深部: 「ユーザーの認知リズムを理解した UX 設計力」「同期・非同期の境界設計の眼」「文化と技術の両面を見渡す視点」

### 本質の質問

- この組織は **何の問いに対する答え**として存在しているか?
- マインドセット・ビジョンを言葉にしたとき、それは**どんな世界観**を含んでいるか?
- 既存商品から**抽象化**すると、組織は本質的に**何が得意**なのか?
- 譲れない一線は、組織が **何を最優先しているか**の証拠になっている。それは何か?

### アウトプット

組織の本質を **3 〜 5 つのケイパビリティ命題**として言語化する。各命題は「組織は X が本質的に得意である」「組織は X を世界に問うている」のような形。

例:

> 組織の本質ケイパビリティ
> 1. ユーザーの認知リズム（集中・中断・復帰）を技術的に理解し、UI に翻訳する力
> 2. 同期・非同期コミュニケーションの境界を設計する眼
> 3. 文化と技術の両面に作用する介入をデザインする視点
> 4. 個人ツールから組織文化への波及をプロダクトで仕掛ける思想
> 5. （ビジョン）「コミュニケーションが苦手な人もコミュニケーションしやすい場づくり」

> 注: 5 番目のような**ペルソナの Pain と直接結びつかないが組織が持っているビジョン**も必ず本質に含めること。これが Synthesis で非自明なマッチを生む鍵。

---

## C. 非自明なマッチポイントの発見

A（ペルソナの根本原因）と B（提供者の本質ケイパビリティ）を突き合わせる。**ここでエージェントの分析力を最大に発揮する**。

### 探し方

3 種類のマッチを探す:

1. **直接マッチ**: 提供者の本質ケイパビリティが、ペルソナの根本原因の構造に直接作用する。例:「同期・非同期の境界設計の眼」 ↔「即時応答強要の文化的構造」。
2. **転用マッチ**: 提供者が別領域で蓄えたケイパビリティが、ペルソナの根本原因に転用できる。例:「コミュニケーション弱者の場作りの知見」が「集中したい人の場作り」に転用できる。
3. **直交マッチ（最も非自明）**: ペルソナ側からは絶対に見えてこないが、提供者側のビジョン・マインドセットから繋がるアプローチ。例: ペルソナは「Slack の通知制御アプリ」を求めているように見えるが、提供者のビジョン「コミュニケーションが苦手な人の場作り」を起点に考えると、「集中しているとき相手は呼びかけてもいい・無視されることに価値がある」という**規範のフレームを変える**プロダクトに到達する。

### マッチごとの提示フォーマット

各マッチは次の構造で言語化する:

```
[マッチ名] わかりやすい呼び名（例: M1. 規範フレームの再定義）

提供者側の武器: [どの本質ケイパビリティから来ているか]
ペルソナ側の対応: [どの Pain / Gain に効くか、または根本原因のどの層に作用するか]
メカニズム: [なぜ・どう繋がるか。1〜3 行で]
タイプ: 直接 / 転用 / 直交
確信度の自覚: [ハルシネーション疑いがあれば明示]
```

### 数

主要な Pain / Gain をカバーする 3 〜 6 案を出すのが目安。多すぎると検証コストが上がるので絞る。**直交マッチを最低 1 つは含める**よう意識する（このスキルの面白さの核）。

### "稼ぎ" は出さない

マッチ案に「これで儲かる」「市場が大きい」のような事業感の根拠を混ぜない。**メカニズムの妥当性**だけで提示する。収益化はステップ 6 で軽く触れる。

## D. ユーザーへの提示と検証（質問シート 006）

エージェントが A / B / C の結果を **1 枚の質問シート** に書き出し、ユーザーがファイル上で各マッチ案を評価する形にする。

### 出力先

```
./docs/value-proposition/0000-[persona-name]/questions/006-synthesis-match-validation.md
```

### シートの構成

冒頭に進捗表示、続いて A・B の分析結果を**読み物として掲載**（設問化しない — ここは分析成果の共有）、最後に C のマッチ案ごとに 1 問を配置する。

```markdown
# 質問シート 006 — 統合の検証（マッチ案評価）

進捗: [✓Customer Profile 確認] [✓提供者の棚卸し] [▶統合] [ ]落とし込み [ ]Fit 確認 [ ]収益化 [ ]HTML 出力

> 回答方法: 各マッチ案について `回答:` 行に A / B / C / D を書き、修正・不採用（B / C）の場合は理由を、D の場合は自由記入欄に代替案・補足を書いてください。複数採用可、空欄不可。

---

- 直接の原因: ...
- 構造的原因: ...
- メタ原因: ...
- 連鎖: ...
- 本質的な軸: ...

2. 同期・非同期コミュニケーションの境界を設計する眼

## C. マッチ案の評価

### M1. 規範フレームの再定義（直交マッチ）

- 提供者側の武器: ビジョン「コミュニケーション弱者の場作り」、ケイパビリティ「文化と技術の両面に作用する介入」
- ペルソナ側の対応: P1 の構造的原因「即時応答の文化的強要」、G2「自律性の保持」
- メカニズム: 受信側の通知制御で対処せず、送信側の規範を上書きする。Pain の構造そのものを変える最も深いレイヤーへの介入。
- タイプ: 直交
- 確信度: 中（メカニズムは強いが、規範の上書きは時間がかかる。実証は要 PoC）

**Q. この案をどう扱いますか?**
- A. 採用（このまま Pain Reliever / Gain Creator に載せる）
- B. 修正して採用（理由・修正内容を D に書く）
- C. 不採用（理由を D に書く）
- D. 自由記入:

回答:

### M2. ...
```

### 発散をシートで求めない理由（重要）

シートに「あなたなら他にどんなマッチを思いつきますか?」といった発散設問を混ぜない。それをやると:

1. ユーザーが「思いつかない」プレッシャーを感じる
2. エージェントが用意した思考の枠に囚われる
3. 質問シートがベースとして機能しなくなる

発散はエージェントが済ませ、シートは**検証専用**に保つ。ユーザーの自由発想は「D. 自由記入」に受け皿を用意しておけば十分。

### ユーザー応答後

シートを読み取り、採用されたマッチ案を `state.json` の `synthesis.user_validated_matches` に記録し、ステップ 4（落とし込み）に進む。

## エージェントの振る舞い

- A と B はそれぞれ独立に深く掘る。**A を B 寄りに、B を A 寄りに歪めない**。
- マッチ案を提示するときは、各案の**メカニズム**を 1 〜 3 行で必ず言語化する。表面的なキーワードの一致で繋げない。
- **直交マッチを最低 1 つ**は含める。このスキルの面白さの核。
- ハルシネーション疑いがあれば自分から明示する。「この案は組織の棚卸しから直接は出ていないが、ビジョンから推論した。確信度は中」のように。
- ユーザーの修正・不採用理由は、後段の Pain Reliever / Gain Creator の精度に効くので、**理由を聞き出す**ことを意識する。
