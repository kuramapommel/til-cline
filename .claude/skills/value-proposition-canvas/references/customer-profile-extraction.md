# Customer Profile 抽出のヒューリスティクス

入力されたペルソナ像から、Customer Profile（Customer Jobs / Pains / Gains）を抽出する際の手引き。**機械的に当てはめず、出発点としての叩き台**として扱うこと。最終的にはユーザー確認シートで補正してもらう前提。

## 入力形式の判定

入力は次のいずれもありうる:

1. **`persona` スキル形式の `customer-map.md`** — 5 観点（誰／コンテキスト／モチベーション／障害／アウトカム）が見出しとして明示されている
2. **自由フォーマットの `.md`** — Notion・社内 Wiki・口頭メモなどから貼られた構造化されていない文書
3. **テキスト貼り付け** — 1 段落の口頭説明レベル

**形式判定はしない**。すべて「書かれている情報から要素を読み取る」というスタンスで処理する。`persona` 形式かどうかを判別して挙動を分けない。

## 5 観点 → Customer Profile 3 ブロックへのマッピング

入力に 5 観点相当の情報があれば、次のように再マッピングする。**1 観点 = 1 ブロックではなく、複数観点を組み合わせて各ブロックを構成する**点に注意。

### Customer Jobs ← コンテキスト + モチベーション + アウトカム

- **コンテキスト**にある日常の活動・繰り返している場面 → 機能的 Job の素材
- **モチベーション**にある「達成したい状態」「報酬と感じるもの」 → 情緒的・社会的 Job の素材
- **アウトカム**にある「人生の豊かさが向上する」要素 → 大きめの Job（人生レベルの目標）

例: コンテキストに「毎日の細切れ MTG の合間にコードを書く」、モチベーションに「自律性が報酬」、アウトカムに「集中時間を確保して挑戦できる」とあれば、Customer Jobs として:
- （機能的）細切れの時間でも集中して実装を進める
- （情緒的）自分のペースを保って働く
- （アウトカム的）技術的にチャレンジングな実装に臆さず取り組める状態になる

### Pains ← 障害

- **障害**は基本的にそのまま Pains の素材になる
- 障害の **メカニズム**（なぜそれが Pain なのか）まで読み取って書く

例: 障害に「Slack の割り込みが集中を断ち切る」「半集中状態で疲弊」とあれば、Pains として:
- 集中状態を断ち切られる（Slack メンション・急な依頼）
- 「次に来るかも」という半集中状態が続いて疲弊する
- 計画していた作業時間を侵食され、自分のリズムが崩れる

### Gains ← モチベーション + アウトカム

- **モチベーション**は「日々の小さな報酬・嬉しさ」 → 期待する Gain
- **アウトカム**は「本質的な価値・大きな獲得」 → 望ましい Gain・想定外の Gain

例: モチベーションに「PR がスムーズにマージされる完遂感」「同僚から感謝される」、アウトカムに「待ちからの解放」「キャリアの積み重ね」とあれば、Gains として:
- 完遂感（難しい実装も気持ちよく終わる）
- 同僚から「助かった」と感謝される実感
- 誰かを待つストレスから解放され、自分のペースで仕事を完結できる
- 「あの時期は充実していた」と振り返れる積み重ね

## 自由フォーマット入力の場合

5 観点が明示されていない場合、本文を読み解いて次を抽出する:

- **動詞句で書かれている顧客の行動・目的** → Job の候補
- **「困っている」「面倒」「不安」「リスク」「避けたい」** などのネガティブ表現 → Pain の候補
- **「したい」「欲しい」「嬉しい」「楽になる」** などのポジティブ表現 → Gain の候補

属性情報（年齢・職業・居住地）は Customer Profile には直接書かない。Job/Pain/Gain の表現を**現実的にする**ためのコンテキスト情報として使う（例: 「30 代エンジニア」なら、Job の表現は技術的なリアリティを持たせる）。

## ハルシネーション抑止

書かれていないことは**勝手に補完しない**。例えば:

- ペルソナに「家族構成」が書かれていないのに「子育てでバタバタしている」と書く ❌
- ペルソナに収入が書かれていないのに「節約志向」と書く ❌
- ペルソナに居住地が書かれていないのに「都市部の通勤ストレス」と書く ❌

足りない情報は空欄のまま残し、ステップ 1 の確認シートでユーザーに「ここは空欄ですが、追加できる Pain はありますか?」と聞く。

## 確認シートでの提示の仕方

抽出結果は確認シート（`001-customer-profile-check.md`）で**そのまま提示せず**、各項目に「採用 / 修正 / 削除 / 追加」のチェックを促す。

```markdown
# 質問シート 001 — Customer Profile（顧客側）の確認

進捗: [▶Customer Profile 確認] [ ]提供者の武器棚卸し [ ]Gain Creators 発想 [ ]Pain Relievers 発想 [ ]Products and Services 落とし込み （フェーズ 1/5）

> ご提供いただいたペルソナ像から、Customer Jobs / Pains / Gains を以下のように抽出しました。
> 各項目について、「✓ 採用 / 修正案を D に書く / × 削除」のいずれかを選んでください。
> 末尾の Q-add に「追加したい項目」があれば自由記入してください。

## Customer Jobs（顧客が片付けたい用事）

### J1. 細切れの時間でも集中して実装を進める
- A. ✓ 採用
- B. 修正したい（Dに修正案）
- C. × 削除
- D. 自由記入:

回答: 

### J2. 自分のペースを保って働く
- A. ✓ 採用
...

### Q-add. 追加したい Customer Jobs があれば

自由記入:

回答:
```

このフォーマットなら、ユーザーが「素材は良いがこの表現は違う」と思ったときに修正コストが低い。
