# バリュープロポジションキャンバス ワークフロー（価値提案ビルダー）

このスキルは、すでに描かれている **ペルソナ像（顧客像）** を出発点に、バリュープロポジションキャンバスを対話的に組み上げ、最終的に 1 枚の HTML（キャンバス＋ UX ストーリーボード＋確信度・検証・収益化セクション）として出力するためのワークフローです。エージェント（あなた）はファシリテーターであると同時に**分析者**でもあり、顧客側の抽出と、提供者側の純粋な棚卸し、そして両者の非自明なマッチングを担います。

> **本ドキュメント内部の表記について（重要）**
>
> SKILL.md・references/ の内部記述ではトークン削減のため「VPC」と略してよい。ただし**ユーザーに見える出力（質問シートのタイトル・本文、HTML 内の文言、エージェントから user へのメッセージ）では、必ず「バリュープロポジションキャンバス」と日本語で書く**。略称は絶対に使わないこと（クラウドインフラの VPC と混同されるため）。本ドキュメント以下、内部表記としてのみ VPC を使う。

## このスキルが本来やるべき向き（重要）

このスキルの**価値の核**は次の向きで進めることにある:

> **提供者側のことを（純粋に）聞く → エージェントが提供者の本質的な武器とペルソナの根本原因の双方を分析 → 両者の非自明なマッチを発見してユーザーに提案する**

逆向き（**ペルソナの Pain/Gain の解消に繋がりそうなことを先回りして聞く → そのまま Pain/Gain にマッピングする**）は **絶対に避ける**。質問の時点でペルソナ起点のフィルタや誘導ヒントをかけると、提供者側の回答空間が「わかりやすい解」に収縮し、本来そこにあるはずの**ペルソナと直交するが強力な武器**（例: 組織のビジョンや暗黙のケイパビリティ）が surface しなくなる。アドホックなプロダクトデザインに陥り、このスキルの一番おいしい工程が短絡される。

エージェントはペルソナを「**後段の統合のために頭の中に保持するだけ**」とし、**提供者への質問内容には一切漏らさない**こと。先回りせず、広く・生のまま集める。マッチングはエージェントが分析の重労働を担う。

## このスキルの価値

- ペルソナ起点では出てこない**提供者側の本質的な武器**を、ペルソナ非依存の棚卸しで surface させる。
- ペルソナの**根本原因**と提供者の**本質的なケイパビリティ**を、エージェントが構造化・深掘りして**非自明なマッチ**を発見する。
- 噛み合い（Fit）を論理的に点検し、不整合をユーザー本人に気づかせる。
- 1 枚の HTML として可視化し、チーム共有や後工程（プロトタイプ・PoC）の起点として使える成果物にする。

## 全工程で守る基本姿勢

1. **「仮説」スタンスを保つ。** ここで立てるのはあくまで「価値提案仮説」。出力文章でも「〜と想定される」「〜が効きうる」のように仮説のニュアンスを残し、断定しない。
2. **入力されたペルソナにつられすぎない。** 粗い入力でも詳細な入力でも、書かれていない情報を勝手に外挿しない。
3. **提供者への質問はペルソナ色付け禁止。** 「武器候補」「ペルソナの Gain に効きそうな観点」のような誘導を質問のリード文・選択肢に混ぜない。回答例は**提供者の実体として多様にありうる選択肢**から作る。
4. **提供者は組織。** 個人を主語にしない（「〇〇さんなら頼める」のような問いは禁止）。提供者は企業・組織・最低でもプロダクトチームを想定し、組織の総合力を引き出す。
5. **ユーザーが答えに迷う項目は無理に埋めない。** 空欄は弱い仮説として残し、検証ポイントに送る。
6. **スコープは 1 ペルソナ = 1 キャンバス。**

## 内部方針（重要・ユーザーには直接説明しないこと）

以下は VPC 構築の挙動を縛る方針だが、**ユーザー向けの説明文・メッセージとしては出さないこと**。「事業感を抑えている」と明かすと意図がメタ認知され、対話の自然さが損なわれる。

- 棚卸し・統合フェーズで、**事業感（収益性・価格・市場規模・競合シェアといった "稼ぎ" の問い）を表面化させない**。発散・創造性・本質を優先する。
- 収益化は最後（後段ステップ）にエージェントの推測案として軽く触れるだけでよい。
- 質問の表面では「儲かりますか?」「価格はいくらにしますか?」のような問いを出さない。

## スコープ

- 1 ペルソナ → 1 キャンバス。
- 価格設計・市場規模算定・競合分析・事業計画書はスコープ外。HTML 末尾の「収益化仮説」枠でエージェントの軽い推測を提示するに留める。
- ペルソナ像の作成自体はスコープ外。粒度が薄い場合はステップ 0 で具体化を促すが、本スキル内ではペルソナ作成までは行わない。

## 入力形式

次のいずれの形でも受け付けること。特定フォーマットを前提にしない。

- テキストの貼り付け
- ファイルパス指定（顧客マップ `.md`、自由フォーマットの `.md`、Notion からエクスポートされた `.md` など）
- インラインの口頭説明

入力形式の判定はせず、書かれている情報から「誰／コンテキスト／モチベーション／障害／アウトカム」相当の要素を読み取って、Customer Profile（Customer Jobs / Pains / Gains）に再マッピングする。

## 対話フロー

### ステップ 0 — ペルソナの粒度ゲート（ソフトゲート）

入力された顧客像が「漠然とした思いつきレベル」など具体化が不足している場合、**ハードに止めずに**次を行う。

- 顧客像が薄いと VPC の精度が落ちうる旨を伝える。
- 「先に顧客像を具体化してから進める / このまま進める」をユーザーにヒアリングし、判断を委ねる。
- 具体化を勧める場合の案内は **汎用的な言い回し**にとどめ、特定スキル名を名指ししない。

**粒度の目安:** 単なる属性（年齢・職業）だけでなく、次の 3 観点のうち **少なくとも 2 つ**が言語化されていれば概ね通過。
1. 具体的な状況・文脈
2. 達成したいこと／抱えている課題
3. その背景にある動機

1 つ以下、または「〇〇な人」程度の一行表現なら薄いと判断してヒアリングを促す。

### ステップ 1 — Customer Profile（顧客側）の抽出と確認

入力から顧客側 3 ブロックを抽出する。**機械的に当てはめず、出発点としての叩き台**として扱うこと。詳しいマッピング手引きは `references/customer-profile-extraction.md`。

- **Customer Jobs** ← コンテキスト＋モチベーション／アウトカムから導出
- **Pains** ← 障害から導出
- **Gains** ← モチベーション＋アウトカムから導出

抽出したら **いきなり確定せず、最初の質問シート（`001-customer-profile-check.md`）でユーザーに提示し、補正・追加・削除してもらう**。

### ステップ 2 — 提供者側の純粋な棚卸し（ペルソナ非依存）

提供者の「いまの実体」をそれ自体として広く引き出す。**質問にペルソナの色を一切持ち込まない**。エージェントはペルソナを頭の中に保持するだけで、質問のリード文・選択肢・誘導ヒントには絶対に漏らさない。

詳細は `references/value-map-elicitation.md`。

**シート構成（3 枚）:**

1. **シート 002 — マインドセット・ビジョン・ポテンシャル**: 組織が何を信じ、何を実現したくて、どんな伸びしろを持っているか。**最も重要なシート**。ここが薄いと後段の統合が「ありがちな解」に流れがち。
2. **シート 003 — 資産・ノウハウ・ナレッジ・事業ドメイン**: 組織が持っている／使えるリソース、暗黙知含めた知識、専門領域。
3. **シート 004 — 既存の商品・製品・サービス・差別化要素**: 既に提供しているもの／過去に提供したもの、組織として「これは任せてほしい」と言える領域・総合力、撤退・失敗の経験。

**主語は組織。** 「あなた」ではなく「あなたの組織／チームは」「この組織として」と問う。「〇〇さんなら頼める」のような個人主語の問いは禁止。組織の総合力こそが武器でありポテンシャル。

**回答例の作り方:** ペルソナと一切結び付けず、**提供者の実体として多様にありうる選択肢**から作る。「ありがちな選択肢」と「あえて違う角度の選択肢」を混ぜて思考を揺さぶる。例: 組織のビジョンを問うとき、ペルソナの集中課題に紐付けた選択肢ではなく、組織として持ちうる多様なビジョン候補（業界変革／カテゴリ創造／ニッチ深耕／大衆化／文化形成 など）を出す。

ユーザーが答えに迷う項目は空欄でよい。空欄も価値ある情報として保持。

### ステップ 3 — エージェント主導の統合（Synthesis）

このスキルの**核・腕の見せどころ**。両側が独立して揃ってから、エージェントが分析の重労働を担う。質問シートにせず、**チャットでの分析レポート＋マッチング案提示 → ユーザー検証／補正**の形で進める。詳細は `references/synthesis.md`。

**3 つの分析を行う:**

1. **ペルソナの根本原因分析**: 表層の Pain の下にある構造・因果まで掘る。「なぜこのペルソナはこの状況に陥っているのか」をメカニズムとして言語化する。
2. **提供者の本質分析**: 表層の商品ラインナップと、深部の**転用可能なケイパビリティ**を切り分ける。組織が本質的に活かせる武器を抽出する。マインドセット・ビジョンも武器として扱う。
3. **非自明なマッチポイントの発見**: ペルソナの根本原因と提供者の本質武器を突き合わせ、**ペルソナ側からは見えないが提供者側から繋がる**マッチを構造的に探す。マッチごとにメカニズム（なぜ繋がるか・どう作用するか）を言語化する。

**提示の形:** チャットで「分析レポート」を出す。各マッチ案には次を添える:
- 提供者側の武器（どの棚卸しから来ているか）
- ペルソナの Pain / Gain（どこに効くか）
- メカニズム（なぜ・どう繋がるか）
- 確信度の自覚（ハルシネーションの疑いがあれば明示）

**ユーザーの役割:** マッチ案を読んで「採用 / 修正 / 不採用」を返してもらう。修正・不採用には理由をもらえると後段に活きる。エージェントは決め打たず、複数マッチ案の中から複数採用してよい。

統合の発想フェーズを質問シートにしない理由: 質問シートに「マッチ案」を載せると、選択肢が回答者にバイアスを与え、思考を狭める。**統合はエージェントの分析責任**として、検証・補正だけをユーザーに渡す。

### ステップ 4 — Products and Services への落とし込み

採用されたマッチ案（Gain Creators / Pain Relievers の確定版）から、具体的な提供物の形態に落とし込む。

シート **005** を 1 枚出して、形態（アプリ / SaaS / ハードウェア / コンテンツ / コーチング / コミュニティ / 書籍 / 講座 / 統合サービス）と、単発／継続、タッチポイントを問う。事業計画には踏み込まない。

### ステップ 5 — Fit の最終確認

統合フェーズでメカニズム言語化は済んでいるはずなので、ここでは **最終的な整合確認**を行う。詳細は `references/fit-check.md`。

- 採用された Pain Reliever / Gain Creator がすべての主要 Pain / Gain をカバーしているか
- 提供者側に「浮いた要素」（顧客側のどこにも結びつかないもの）はないか
- 「補強する／検証ポイント送りにする／そのまま進む」をユーザーに選んでもらう

### ステップ 6 — 収益化の仮説（軽め）

エージェントとして収益化の推測案を 2 〜 4 案提示する。**主役ではなく軽いレコメンド**。形態（買い切り／定額／従量／フリーミアム／広告／コミュニティ会費／プロフェッショナルサービス など）と、その案がこのペルソナにフィットしうる理由を 1 〜 2 行で添える。質問シートで深掘りしない。

### ステップ 7 — 成果物の出力

成果物 HTML を生成する。

- 出力先: `./docs/value-proposition/0000-[persona-name]/value-proposition.html`
- 単一ファイル自己完結（CSS・SVG インライン、外部依存なし）。
- スキャフォールド `assets/canvas-template.html` をベースに、6 ブロックのプレースホルダを埋め、UX ストーリーボード SVG とフッタの「確信度・検証・収益化」セクションを追記する。
- 詳細仕様は本ドキュメント末尾の「成果物 HTML の仕様」セクション。

## 質問シートの仕様

質問は会話メッセージとしてではなく、**markdown の質問シート（一時ファイル）**として書き出し、ユーザーがファイルを直接編集して回答する運用にする。

> **質問シートを使うフェーズは限定的:** ステップ 1（顧客側確認）、ステップ 2（提供者側棚卸し 3 枚）、ステップ 4（Products and Services 落とし込み 1 枚）。ステップ 3 の統合・ステップ 5 の Fit 確認・ステップ 6 の収益化はチャットで行う（質問シートにすると誘導が入るため）。

### 出力先

```
./docs/value-proposition/0000-[persona-name]/questions/000-[question-summary-title].md
```

### シートのフォーマット

各設問は **A 〜 C の回答例 + D 自由記入** の形にし、回答負担を下げる。**回答例の作り方は重要**: 

- ステップ 1（顧客側確認）の回答例は「採用 / 修正 / 削除」の選択。
- **ステップ 2（提供者棚卸し）の回答例は、ペルソナと一切結び付けない**。提供者の実体として多様にありうる選択肢から作る。「ありがちな選択肢」と「あえて違う角度の選択肢」を混ぜて思考を揺さぶる。
- ステップ 4（落とし込み）の回答例は具体的な形態の候補。

各シート冒頭に **進捗（現在フェーズ／残り）**を必ず表示する。

```markdown
# 質問シート 002 — マインドセット・ビジョン・ポテンシャル

進捗: [✓Customer Profile 確認] [▶提供者の棚卸し] [ ]統合 [ ]落とし込み [ ]Fit 確認 [ ]収益化 [ ]HTML 出力 （シート 1/3）

> 回答方法: 各設問の `回答:` 行に A / B / C / D を書き、D を選んだ場合は自由記入欄に追記してください。複数選択可、空欄も OK です（空欄は「仮説の弱い箇所」として検証ポイントに送ります）。

## Q1. ......
- A. ......
- B. ......
- C. ......
- D. 自由記入:

回答:
```

### 疲労サイン時の振る舞い

ユーザーが疲れているサインを感じたら（短い回答が続く・「もう十分」のような言葉が出る・回答の質が落ちる）、その時点で**中間出力する**ことを提案してよい。残りは弱い仮説として HTML に載せ、検証ポイントに送る。

## 成果物 HTML の仕様

**1 枚の HTML に、バリュープロポジションキャンバスと UX ストーリーボードを同梱**する。

### 技術要件

- **単一ファイルで自己完結**: CSS・SVG はインライン、外部依存なし。
- 文字コード UTF-8、`<html lang="ja">`。

### (A) バリュープロポジションキャンバス部

- **2 カラムレイアウト**: 左カラム = 提供者側 Value Map、右カラム = 顧客側 Customer Profile。
- **各カラム内の 3 ブロックは縦積み**にする（横並びにしない。横並びは縦に細くなり読みづらい）。
- 提供者側の縦並び順: **製品・サービス → 悩みを和らげるもの → うれしさを生み出すもの**（または上から Gain Creators / Pain Relievers / Products and Services でも可。決めたら一貫させる）。
- 顧客側の縦並び順: **顧客の用事 → うれしさ → 悩み**（同じく一貫させる）。
- 各ブロックには見出し（日本語＋英語併記）と項目リスト、**対応タグ**（`P1` / `P1·P2` / `G2` 等）で対応関係を可視化する。
- Strategyzer の公式テンプレ画像・ロゴ・QR コード・タグライン等は流用しない。オリジナルデザインで描く。
- スキャフォールド `assets/canvas-template.html` を基点に、各プレースホルダにブロック内容を流し込む。

### (B) UX ストーリーボード部

- 4 コマ漫画／絵コンテ／ビフォー・アフター 2 コマなど、**製品・サービスのイメージに合わせて毎回自由に**表現する。詳しくは `references/storyboard-guide.md`。

### (C) 確信度・検証ポイント・収益化仮説セクション（軽い枠）

HTML 末尾に、`persona` スキルの `Confidence & Verification` 程度の重みで載せる。

1. **確信度（★）**: VPC 全体で 1 つ。例: ★★★☆☆。1 〜 2 行の根拠付き。
2. **検証ポイント**: インタビューや PoC で確かめるべき不確実な点を箇条書き。
3. **収益化仮説**: 2 〜 4 案、各 1 〜 2 行の理由付き。

### デザインテイスト

- 落ち着いた配色、十分な余白、可読性重視。
- ペルソナの雰囲気に合わせて配色を微調整してよい（ただしキャンバスの構造可読性を最優先）。

## 進行中のお作法

- 各ステップの区切りで進捗（フェーズ／シート）を必ず示す。
- ユーザーの回答が読みにくい・解釈に迷うときは、勝手に補完せず**確認質問**を返す。
- 「これ以上聞かれても答えられない」と感じさせない: 疲れているサインを感じたら**いったん中間出力**し、残りは検証ポイントに回す提案をしてよい。
- 不明点・前提が立てられない・複数の選択肢があって判断に迷うときは、想像で進めずに作業を止めて質問する。
- **略称ルール**を最後まで守る: ユーザーに見える表記では「バリュープロポジションキャンバス」と日本語で書く。「VPC」とは絶対に書かない。
- **ペルソナ非依存ルール**: ステップ 2 の質問シートにペルソナ由来のヒント・武器候補・誘導は一切混ぜない。先回りせず、生のまま集める。
