# infra-improvement-todo ワークフロー

## このスキルの目的

スキル **infra-architecture-review** の評価レポート（または同等のレビュー結果・人手のレビューメモ）を入力に受け取り、
**有限で・完了可能で・検証可能な改善 TODO リスト**へ変換する。

評価結果を、**「対応しきれば本当に完了する」TODO** に変換することがゴールである。
「改善点の列挙（再レビューのたびに増え、対応しきっても完了にならないリスト）」を作ることは禁止する。

このスキルは Claude 以外のエージェントでも単体で実行できるよう、`SKILL.md` だけで自己完結している。

## 入力と前提

- 主たる入力は、同じ `.claude/skills` 配下のスキル **infra-architecture-review** が出力する評価レポートを想定する。
  ただし、人手のレビュー結果・レビューメモ・チャット上の指摘リストなど、同等の「レビュー結果」であれば何でも動作する。
- このスキルは **評価をやり直さない**。評価（何が問題かの発見・判定）は infra-architecture-review の責務であり、
  このスキルは受け取った指摘を **有限な TODO に変換すること** に専念する。
- 方法論はクラウドベンダ非依存であり、インフラ以外の一般的な「レビュー → 改善計画」にもそのまま一般化できる。
  本文中の「インフラ」「SLO」は、対象ドメインに応じた「対象システム」「定量化された業務優先度」と読み替えてよい。

## このスキルが存在する理由（解決する失敗モード）

「改善して」と頼まれたエージェントの素朴な挙動は、品質カタログ（AWS / Google / Azure などの Well-Architected 等）を当てて
「未充足のベストプラクティス」を全部タスク化することである。

しかし **品質は青天井** であるため、このリストは **対応しても再レビューで再増殖し、永遠に完了しない**。

本スキルの核心は **「開いた品質目標」を「閉じた到達点」に変換すること** である。
Well-Architected の本質（= 文脈依存のトレードオフ + リスクベースの優先度付け + 定期レビュー）を正しく使えば、
完了線は有限になる。品質カタログは「100% 達成すべきチェックリスト」ではなく「アンチパターン回避の点検表」として扱う。

## 必須ワークフロー（この順序を強制する）

以下の手順 1 → 5 を **必ずこの順序で** 実行する。とくに手順 1（完了の定義）を、TODO を列挙する前に書くことが重要である。

### 手順 1: 先に「完了の定義 (Definition of Done)」を書く

TODO を列挙する **前に** 完了条件を書く。次の 2 つの錨で定義し、**「指摘がゼロになること」を完了条件に含めてはならない**。

- **機能的・検証可能な錨**：「実際に動く」こと。
  例：デプロイできる / E2E・smoke テストが通る / 既存テストが緑のまま。
- **有限な品質バー**：「各社 Well-Architected が文書化しているアンチパターンを踏んでおらず、
  かつ非自明な判断すべてを SLO（定量化された業務優先度）で正当化できる」状態。

完了条件は **falsifiable（反証可能）なチェックリスト** として書く。
「指摘がゼロ」「もっと良くできる箇所が無い」を完了条件にしてはならない（それは到達不能であり、完了しない）。

### 手順 2: 各指摘を「下すべき判断」に変換する

「未充足のベストプラクティス」を即タスク化しない。各指摘は必ず以下のどちらかで **終わらせる（閉じる）**。

- (a) **実装すべきブロッカー**、または
- (b) **閉じた判断**（例：「やらない。なぜなら SLO 上、現在の業務優先度では不要だから」）

**「改善できる」だけで判断のない項目を残すことを禁止する。** すべての正当化は SLO に紐づける。
「一般論として良い」だけで SLO に紐づかない項目を残してはならない。

### 手順 3: 環境依存（ダイヤル）と設計（本質）を分ける

- 「dev では入れない」等の **環境依存項目** は、コードから消すのではなく
  **変数化して default を dev 想定にする** のが理想である（prod は値を変えるだけで再設計が不要になる）。
  これは「完了をブロックしない／記録すべき判断」に分類する。
- 環境に依らず **壊れている・正当化できない** 項目だけが、設計上のブロッカーである。

判断のポイント：「prod に持っていくとき、値を変えるだけで済むか？ それとも作り直しが要るか？」
値を変えるだけで済むなら環境依存（ダイヤル）、作り直しが要るなら設計（本質＝ブロッカー候補）。

### 手順 4: バケットに振り分ける（Well-Architected のリスクベース優先度に対応）

各指摘を以下のバケットのいずれかに割り当てる。

- **MUST-FIX（完了をブロック）**：
  「壊れている」項目 + 「SLO で正当化不能な設定」のみ。**小さく保つ。**
- **DECISIONS（選んで記録すれば閉じる）**：
  トレードオフ・環境依存ダイヤル。根拠（SLO）を書けば完了する。
- **APP-SIDE（インフラ完了線の外 / 忘れない用メモ）**：
  評価で「インフラを合わせるのではなくアプリ側を直すべき」と示唆された項目。
  インフラの完了はブロックしないが、忘れないよう独立バケットで記録し、
  **対象・理由・付随するインフラ変更（あれば）** を書く。
- **BACKLOG（完了をブロックしない）**：
  正当だが任意の磨き込み。
- **VERIFICATION**：
  機能的に動くことの確認手順（デプロイ + smoke 等）。
  SLO に観測可能性（オブザーバビリティ）が必要なら、その計測も含める。

### 手順 5: 停止ルールを成果物に明記する

成果物の末尾に、次の趣旨を **必ず明記** する。

> Well-Architected レビューは、その時点の業務コンテキスト（SLO）に対する点検であり、
> 「完璧な終着点」の探索ではない。
> **MUST-FIX が閉じ + VERIFICATION が通り + 各トレードオフが SLO で正当化・記録された時点で、完了を宣言してよい。**
> SLO やワークロードが変わったとき **再度** レビューする（定期レビューの思想）。
> 完了判定として網羅的な再レビューを回さない。
> 以降に出てくる指摘は『未完成のサイン』ではなく『停止すべきサイン』である。

## 各 TODO 項目に必須の要素

すべての TODO 項目は、以下の要素を含むこと。

- **何が問題か**：壊れている / SLO で正当化不能 / トレードオフ、のいずれかを明示する。
- **根拠**：`file:line`、または層をまたいだ証拠（複数ファイル・設定の関係）を示す。
- **帰結アクション**：直す / 判断して記録 / 変数化 / 削除・簡素化 / 延期 / アプリ側へ移送 のいずれか。
- **どのバケットか**：完了をブロックするか否かが分かること。
- **チェックボックス**：`- [ ]` で記述する。

## 出力フォーマット（1 つの Markdown）

成果物は **1 つの Markdown** とし、以下のセクションを **この順序** で含める。

```markdown
# 改善 TODO: <対象名>

## 1. 完了の定義 (Definition of Done)
<!-- falsifiable なチェックリスト。「指摘ゼロ」を含めない。
     機能的・検証可能な錨（動く）と、有限な品質バー（アンチパターン無し + 各判断を SLO で正当化）の 2 本柱で書く。 -->
- [ ] （機能）デプロイでき、smoke / E2E が通る
- [ ] （品質）既知のアンチパターンを踏んでいない
- [ ] （品質）非自明な判断はすべて SLO で正当化・記録済み

## 2. MUST-FIX（完了をブロック）
<!-- 「壊れている」 + 「SLO で正当化不能」のみ。小さく保つ。 -->
- [ ] <問題> / 根拠: <file:line> / アクション: 直す

## 3. DECISIONS（選んで記録すれば閉じる）
<!-- トレードオフ・環境依存ダイヤル。SLO 根拠を書けば完了。 -->
- [ ] <判断事項> / 根拠: <file:line> / 決定: <選択 + SLO 上の理由> / アクション: 記録（必要なら変数化）

## 4. APP-SIDE（インフラ完了線の外 / 忘れない用メモ）
<!-- アプリ側を直すべき項目。インフラ完了はブロックしない。 -->
- [ ] <対象> / 理由: <なぜインフラでなくアプリか> / 付随するインフラ変更: <あれば>

## 5. BACKLOG（完了をブロックしない）
<!-- 正当だが任意の磨き込み。 -->
- [ ] <磨き込み項目> / 根拠: <file:line>

## 6. VERIFICATION（機能確認手順）
<!-- 動くことの確認。SLO に観測可能性が要るなら計測も。 -->
- [ ] <デプロイ / smoke / E2E などの具体手順>

## 7. すでに整合している点（記録）
<!-- 評価で問題なしと確認された点。完了線が有限であることの裏付けとして残す。 -->
- <整合している点>

## 8. 停止ルールの注記
<!-- 手順 5 の文面をそのまま記載する。 -->
```

## ガードレール（絶対に守る）

- 成果物を「改善点リスト / もっと良くできること」として構成しない。
- 重要度（高 / 中 / 低）を **唯一の分類軸** にしない（リスクベースのバケットで分ける）。
- ベストプラクティスカタログを「100% 達成すべきチェックリスト」として扱わない。
- 「何もしない（根拠付き）」「削除する（オーバーエンジニアリング除去）」「アプリ側へ移送」を、
  **正当な帰結** として許可する。
- **機能的・検証可能な錨を必ず含める。**「動かす」と「美しくする」を区別する。
- トレードオフの正当化は必ず SLO に紐づける（「一般論」だけの項目を禁止する）。

## 良い例 / 悪い例（対比）

### 悪い例（開いた TODO・完了しない）

柱別に 🔴🟠🟡 を付けて課題を列挙するだけ。

```
## 信頼性 🔴
- マルチ AZ になっていない
- バックアップが自動化されていない
## セキュリティ 🟠
- IAM ポリシーが広すぎる
- 暗号化が一部未適用
...（以下、品質軸ごとに延々と続く）
```

問題点：完了条件がない。「壊れている」と「最適でない」が混在している。
対応しきっても、再レビューすればまた新しい未充足項目が増殖する。**永遠に完了しない。**

### 良い例（閉じた TODO・完了する）

先頭に「完了の定義」を置き、各項目を判断で閉じる。

```
## 1. 完了の定義
- [ ] (機能) `terraform apply` が通り、smoke (`curl /health` が 200) が緑
- [ ] (品質) 既知のアンチパターン（平文シークレット、0.0.0.0/0 の全開放）が無い
- [ ] (品質) マルチ AZ 等の非自明な判断は SLO で正当化・記録済み

## 2. MUST-FIX（完了をブロック）
- [ ] DB シークレットが平文で variables.tf に存在 / 根拠: variables.tf:42 / アクション: 直す（Secrets Manager 参照へ）

## 3. DECISIONS（選んで記録すれば閉じる）
- [ ] マルチ AZ にするか / 根拠: main.tf:88 / 決定: dev は単一 AZ。SLO は可用性 99%（月 7h 許容）で単一 AZ で足りる。変数化し prod は multi_az=true / アクション: 変数化 + 記録

## 6. VERIFICATION
- [ ] `terraform apply` 成功 → `curl https://.../health` が 200 を返す

## 8. 停止ルールの注記
（停止ルール本文）
```

利点：MUST-FIX を閉じ、E2E（smoke）が通れば **「完了」と宣言できる**。
各項目が判断で閉じており、停止ルールが明記されているため、再レビューで無限増殖しない。

## スコープと連携（まとめ）

- 入力：infra-architecture-review の評価レポートを想定するが、人手のレビュー結果でも動作する。
- 責務：このスキルは **評価し直さない**。受け取った指摘を有限 TODO に変換することに専念する。
- 一般化：方法論はクラウドベンダ非依存であり、インフラ以外の一般的な「レビュー → 改善計画」にも適用できる。
