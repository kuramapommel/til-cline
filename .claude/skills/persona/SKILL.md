# persona ワークフロー（顧客マップ・ペルソナ仮説ビルダー）

このスキルは、**新規プロダクト開発の前段**で「誰のために作るのか」を 1 人分のペルソナ仮説として言語化するための対話ワークフローです。エージェント（あなた）はファシリテーターとなり、スキル利用者（以降「ユーザー」）に質問を重ねて、5 観点の「顧客マップ」を段階的に埋め、最終的にペルソナプロフィール（HTML）まで導出します。

## このスキルの価値

- ユーザーインタビューに挑む**前**の準備として、確度の高いペルソナ仮説を立てる。あわよくばインタビューを不要にできるレベルまで詰めることが理想。
- ユーザー自身は頭の中にメンタルモデルを持っていることが多い。質問を重ねることで、それを引き出して構造化する。

## 全工程で守る基本姿勢

このスキルが扱うのはあくまで「ペルソナ**仮説**」です。次の 2 点を最後まで保ってください。

1. **断定を避け、終始「これは仮説である」というスタンスを保つ。** 出力する文章でも「〜だろう」「〜と想定される」「〜の可能性が高い」のように仮説のニュアンスを残す。「〜である」と断定しない。
2. **入力された初期情報につられない。** 粗すぎる入力（思いつきレベル）でも、緻密な入力（企画書レベル）でも、エージェントはハルシネーションを起こしやすい。**入力情報は「その時点での想定」として受け止め**、勝手に外挿・補完しないこと。書かれていないことは「空欄」のまま残してよい。

ユーザーが答えに迷ったり、自信がない項目は**無理に埋めず空欄**にしてください。空欄であることそのものが「インタビューで検証すべきポイント」になります。

## ペルソナのスコープ

- **必ずプライマリ 1 人**に絞ります。会話の中で「A さんと B さん、どちらもありえる」のような分岐が見え始めたら、**どちらか 1 人を選ぶようファシリテート**してください。複数案が並走するとイメージがブレ、後の意思決定がにぶります。
- セカンダリ・三次以降は今回のスコープ外です（DevOps サイクルを 1 巡してから検討する想定）。

## 顧客マップ＝ 5 つの観点

各観点では「**今回作ろうとしているプロダクトに直接もしくは間接的に影響しそうな情報**」に絞って引き出してください。プロダクトに無関係な雑多な属性まで広げないこと。

1. **誰**: 住所/氏名/年齢/生年月日/星座/血液型/家族構成/趣味趣向/体型/心情 等、個人を特定するに値する直接的なパーソナル情報群。
2. **コンテキスト**: 現在抱えている日常的な状況や背景、日々の出来事、過去の経験など。
3. **モチベーション**: 満足する報酬。金銭的なものに限らず、生活の豊かさや困りごとの解消、承認欲求の充足なども含む。
4. **障害**: 現在抱えている課題・困りごと、将来の不安・懸念など、害となるもの。
5. **アウトカム**: それを得られることで人生の豊かさが向上するなど、本質的な価値を享受できるもの。

## 対話フロー

### ステップ 0 — 起動とプロダクト理解度の確認

引数ありで呼ばれた場合（例: `/persona 単身赴任者向けの〇〇アプリ`）はその内容を出発点に、引数なしの場合はまず次を確認します。

1. **プロダクトの理解度はどのくらいですか?**（次から選んでもらう。選択肢以外の自由回答も可）
   - A. 漠然とした思いつきレベル
   - B. 解きたい課題はある程度見えている
   - C. プロダクトの方向性が固まっており、ラフな企画書がある
   - D. 詳細な企画書・要件がある
2. 上記を踏まえ、**プロダクト概要を粗くてよいので教えてください**。
   - 何のためのプロダクトか / どんな課題を解こうとしているか / 想定している利用シーン、など分かる範囲で。
3. 理解度に応じて伴走の仕方を変えます。
   - **A・B**: 質問数は少なめから開始し、ユーザーのメンタルモデルを引き出すことを優先。発散ぎみで OK。
   - **C・D**: 入力情報に対する**裏取り・反証質問**を多めに織り交ぜ、決め打ちを防ぐ。

### ステップ 1 — 5 観点に「ざっくり」同時回答

プロダクト概要のヒアリングが終わったら、**まず 5 観点すべてに対して「現時点で思い浮かぶ範囲でざっくり」**回答してもらいます（必須ではなく、空欄もあり）。

- 目的: 全体像を一度に俯瞰すると、観点同士の関連が見え、後の段階的詳細化で効率よく深掘りできるため。
- このとき**最初の質問シートを 1 枚**書き出します（後述のフォーマット）。設問は 5 観点に対して各 1〜2 問ずつ、合計 5〜10 問程度で軽め。

### ステップ 2 — 固定順で段階的に詳細化（ただし行きつ戻りつ可）

ざっくり回答が揃ったら、**固定順（誰 → コンテキスト → モチベーション → 障害 → アウトカム）**で観点ごとに深掘りします。

- 各観点で **5〜10 問**を出題。質問はテンプレ固定ではなく、**プロダクト概要とこれまでの回答に応じて毎回動的に生成**する。
- 後の観点の回答次第で前の観点に戻った方がよいことは多い。例えば「アウトカム」を聞いて初めて「誰」の解像度が上がる、など。**ユーザーの回答に応じて柔軟に行き来**してよい。戻る場合は「アウトカムの回答から、『誰』に追加で聞きたいことが出てきました。少し戻ってよいですか?」のように一言断ってから戻る。
- ユーザーが疲弊しないよう、**全体の進捗（現在地と残り）を毎ターン示す**こと。例:
  - `進捗: [✓誰] [▶コンテキスト] [ ]モチベーション [ ]障害 [ ]アウトカム  （観点 2/5、当観点 質問 3/8）`

### ステップ 3 — 整合性チェックと仮説の固化

5 観点が一通り埋まったら、観点をまたいで矛盾・違和感がないかを点検します。矛盾を見つけた場合の扱いは「矛盾・浅い回答の扱い」セクションを参照。

最後に、確信度（★）と検証ポイントを整理して、**顧客マップ（markdown）**と**ペルソナプロフィール（HTML）**を出力します。

## 質問シートの仕様

質問は会話のメッセージとしてではなく、**markdown の質問シート（一時ファイル）**として書き出し、ユーザーがそのファイルを直接編集して回答してもらう運用にします。やり取りの中で**必要都度、複数枚**出力されるイメージです。

### 出力先

```
./docs/persona/0000-[persona-name]/questions/000-[question-summary-title].md
```

- `0000` はペルソナディレクトリの 4 桁ゼロ埋め連番（後述）。
- `000` は質問シートの 3 桁ゼロ埋め連番。出した順に 001, 002, ... と増やす。
- `[question-summary-title]` はその回の質問のテーマを表す短いスラッグ（例: `rough-overview`、`who-deepdive`、`context-deepdive`）。

### シートのフォーマット

各設問は **A〜C の回答例 + D 自由記入** の形にし、回答負担を下げます。回答例は固定文ではなく**プロダクト概要と既出の回答に応じて動的に**作ります。「ありがちな選択肢」と「あえて違う角度の選択肢」を混ぜて、ユーザーの思考を揺さぶれるとなお良いです。

```markdown
# 質問シート 001 — ざっくり全体像
進捗: [▶ざっくり全体像] [ ]誰 [ ]コンテキスト [ ]モチベーション [ ]障害 [ ]アウトカム

> 回答方法: 各設問の `回答:` の行に A / B / C / D を書き、D を選んだ場合は自由記入欄に追記してください。複数選択可、空欄も OK です。

## Q1. 想定するメインユーザーの年齢層は？（観点: 誰）
- A. 20 代前半（学生〜社会人なりたて）
- B. 30〜40 代（働き盛り）
- C. 50 代以上（ベテラン層）
- D. 自由記入: 

回答: 

## Q2. ...
```

シートをユーザーに渡したら、「シートに書き終わったら教えてください」と一言添えて待つ。回答後、シートを読み取ってから次のステップに進みます。

## 矛盾・浅い回答の扱い（6 段階トレードオフ）

ユーザーの回答が浅い、または観点をまたいで矛盾している場合、多くのケースで**ユーザーはトレードオフの両方を大切に思っている**、もしくは矛盾に気づいていません。

攻撃的にならないよう注意しつつ、矛盾点をやさしく指摘してください。例:

> 「Q3 では『手早く済ませたい』、Q7 では『丁寧に向き合いたい』とありました。両方大切にされているのだと思いますが、もしどちらかをプロダクトの軸に置くとしたら、どちらに寄せたいですか?」

そして次の **6 段階のトレードオフ尺度**で重み付けしてもらいます。

```
[1] とても A  [2] A  [3] やや A  [4] やや B  [5] B  [6] とても B
```

### 重要: 敗側の情報も捨てない

重み付けで採用しなかった側（敗側）の情報も**捨てずに**、顧客マップの該当観点に「**補足 / 留意点**」として残してください。後続のコーディングエージェントが開発を続ける際、敗側の情報が欠落すると判断材料が減ってしまうためです。

例（顧客マップ内の記載）:
```markdown
## 障害
- 近くのクリニックは予約がいっぱいで待ちが発生する（主軸）

### 補足 / 留意点
- 「丁寧な診察を受けたい」というニーズもトレードオフの敗側として存在
  （Q7 で『手早さ』を 6 段階の [2] A として採用したが、敗側の『丁寧さ』も視野に入れておく）
```

## 状態管理（中断・再開）

進行状況を**ファイルに保存**し、別セッションでも続きから再開できるようにします。

### 保存先

```
./docs/persona/0000-[persona-name]/state.json
```

### 保存する内容（例）

```json
{
  "persona_name": "tanaka-taro",
  "product_summary": "単身赴任者向けの〇〇アプリ",
  "understanding_level": "B",
  "current_step": "deepdive",
  "current_aspect": "context",
  "current_question_count": 3,
  "answered_aspects": ["who"],
  "tradeoffs": [
    {"a": "手早さ", "b": "丁寧さ", "winner": "a", "weight": 2, "memo": "Q7"}
  ],
  "open_questions": [],
  "questionnaire_files": ["questions/001-rough-overview.md", "questions/002-who-deepdive.md"]
}
```

スキル起動時に**まず `./docs/persona/` 配下に既存のディレクトリがないか確認**し、進行中のものがあれば「`0001-tanaka-taro` の作業が途中のようです。続きから再開しますか?（新しく始める場合は別途その旨指示してください）」と提案してください。

## 成果物・ファイル構成

出力ルート:

```
./docs/persona/0000-[persona-name]/
```

- `0000` は **4 桁ゼロ埋めの連番**。`./docs/persona/` 配下の既存ディレクトリを `ls` などで確認し、最大値 + 1 で採番する。プロダクト名はディレクトリ名に含めない。
- `[persona-name]` は導出したペルソナ名のスラッグ（例: `tanaka-taro`）。仮で進める場合は `tbd` などで開始し、ペルソナ名が固まった段階で `git mv` で改名してよい。
- バージョン管理（v1, v2 等）・日付付与は不要。

ディレクトリ内訳:

```
./docs/persona/0000-[persona-name]/
├── state.json                                  # 進行状況（中断・再開用）
├── questions/                                  # 質問シート（一時ファイル・複数枚）
│   ├── 001-rough-overview.md
│   ├── 002-who-deepdive.md
│   └── ...
├── customer-map.md                             # 顧客マップ（最終）
└── persona-profile.html                        # ペルソナプロフィール（最終）
```

質問シートは「一時ファイル」扱いですが、**自動削除はしません**。残すか削除するかはユーザーに委ねてください（後で見返したい人もいるため）。

## 顧客マップ（最終 markdown）の構成

```markdown
# 顧客マップ — [ペルソナ名]

> これは「ペルソナ仮説」です。インタビュー等で検証することを前提としてください。

## 誰
- ...

### 補足 / 留意点
- ...

## コンテキスト
- ...

### 補足 / 留意点
- ...

## モチベーション
...

## 障害
...

## アウトカム
...

---

## 確信度

★★★☆☆（ペルソナ全体で 1 つ。詳細は後述）

## 検証ポイント（インタビューで確かめるべき不確実な点）

1. ...
2. ...
3. ...
```

## 確信度・検証ポイント セクション

- **確信度**は**ペルソナ全体で 1 つ**、**★（星）表現**（例: ★★★☆☆ は 5 段階中 3）。なぜその水準と判断したかも 1〜2 行で添える。
- **検証ポイント**は「インタビューで確かめるべき不確実な点」を箇条書きで列挙。空欄として残した項目、6 段階トレードオフで僅差だった項目、エージェント側で「ハルシネーションかも」と疑いがあった項目を中心に挙げる。
- このセクションは **顧客マップ（markdown）と HTML の両方に掲載**します。

## ペルソナプロフィール（最終 HTML）の仕様

エージェントが顧客マップを考察して導出したペルソナ像を、**著名人のプロフィールページ**のように表現する HTML を出力します。

### 技術的要件

- **単一ファイルで自己完結**: CSS・SVG はインラインで書き、外部依存（CDN・画像ファイル等）を持たない。ローカルでブラウザに開くだけで完成形が見えること。
- 文字コードは UTF-8、`<html lang="ja">`。

### 含める要素

Wikipedia の著名人ページに相当するもの:

1. **氏名**（仮の日本人氏名でよい。実在の有名人と被らないように。実在の有名人と被ると価値観や経歴が混ざり込んでしまうので注意。）
2. **キャッチコピー**（ペルソナ像を一言で表すコピー）
3. **基本プロフィール表**: 年齢 / 居住地 / 職業 / 家族構成 / 趣味 など、5 観点の「誰」から導出した属性。
4. **経歴 / ストーリー文**: コンテキストから導出した日常や過去経験を、地の文として読み物風に。
5. **アバター**: **インライン SVG** で生成。年齢・性別・雰囲気を反映するが、**丸・四角・線・色で表現できるかなりデフォルメされたもの**でよい（写実性は不要、イメージが伝わればよい）。ユーザーから画像が用意されている場合は `<img>` でのプレースホルダ差し替えを許容する。

Wikipedia には無いが本目的に重要な要素:

6. **ニーズ・課題の図解**: モチベーション / 障害 / アウトカムの関係を SVG で図解する（例: 中央に「障害」、両脇に「モチベーション」と「アウトカム」を配し矢印でつなぐ等）。文章だけでなく図で関係性が見えるようにする。
7. **確信度・検証ポイント セクション**: 顧客マップに載せたものと同じ内容を掲載。

### デザインテイスト

**アーティスト・俳優の事務所公式サイトのプロフィールページ**のような見た目に。具体的には:

- 大きめの余白、洗練されたタイポグラフィ、上品な配色。
- ヒーロー領域（アバター + 氏名 + キャッチコピー）を上部に大きく。
- セクションは横幅を抑えた読み物レイアウト（max-width 720〜960px 程度）。

**配色・雰囲気は導出したペルソナ像のイメージから導出**してください。例:

- 落ち着いた中年男性 → ネイビー / グレー / ベージュ系
- 若い女性クリエイター → ペールトーン / アクセントにビビッドカラー
- 外向的なアスリート → 高彩度・コントラスト強め

「無難な汎用テンプレ」にならないように、ペルソナごとに視覚的に変化を付けてください。

## スコープ外

- 添付資料にあった「**取り消し線 / ピンク色**」表現（ユーザーインタビューで仮説が覆された痕跡）は、今回スコープ外です。実装しないでください。今回のスキルが扱うのは「インタビュー前の仮説構築」までです。

## 進行中のお作法

- 各ステップの区切りで、進捗（観点 N/5、当観点 質問 M/X）を必ず示す。
- ユーザーの回答が読みにくい/解釈に迷うときは、勝手に補完せず**確認質問**を返す。
- 「これ以上聞かれても答えられない」と感じさせない: ユーザーが疲れているサインを感じたら、**その時点でいったん顧客マップを中間出力**し、残りは検証ポイントに回す提案をしてよい。
- 何か不明点・前提が立てられない・複数の選択肢があって判断に迷うことがあれば、想像で進めずに作業を止めて質問してください。
