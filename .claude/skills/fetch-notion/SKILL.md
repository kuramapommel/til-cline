# fetch-notion ワークフロー

Notion の仕様ページを取得し、構造化された形でユーザーに提示する。
取得と解釈に専念し、Issue 作成・ファイル書き出し・Notion への書き込み・コードベースの探索は行わない。

## ステップ 1 — 引数を確認する

**引数なしで呼ばれた場合**
→「取得したい Notion ページの URL、または探したい仕様のキーワードを教えてください。」と聞き返す。

**引数が Notion URL の場合**（`https://www.notion.so/` または `https://notion.so/` で始まる）
→ **URL 指定モード**として、そのページを直接取得する（ステップ 3-A へ）。

**引数が URL 以外のキーワード・自然言語の要求の場合**
→ **検索モード**として、ワークスペースを検索して候補を絞り込む（ステップ 3-B へ）。

## ステップ 2 — Notion MCP の利用可能性を確認する

Notion MCP が利用可能かどうかを確認する。
利用可能かどうかは、セッション開始時に `mcp__notionApi__*` 形式のツールが存在するかで判断できる。

**Notion MCP が使えない場合**は、その旨をユーザーに伝えて作業を止める。
> 「Notion MCP が利用できないため、コンテンツを取得できません。
>  Notion MCP の設定については管理者に確認してください。」

## ステップ 3-A — URL 指定モード：ページコンテンツを直接取得する

指定された URL から Page ID を抽出し、Notion MCP でそのページのコンテンツを取得する。

Notion URL の典型的な形式：
- `https://www.notion.so/<page-title>-<page_id>`
- `https://www.notion.so/<workspace>/<page_id>`
- `https://notion.so/<page_id>`

取得できる情報の例：ページタイトル、本文ブロック、データベースプロパティなど。

取得後はステップ 4（解釈・構造化）へ進む。

## ステップ 3-B — 検索モード：候補を提示してからページを取得する

**検索は二段構えで行う。**
コンテキスト消費を抑えるため、全文を一括取得するのではなく、まず候補一覧を絞り込む。

### 3-B-1. 検索を実行して候補一覧を取得する

Notion MCP の検索ツールを使い、ユーザーのキーワード・要求に合致するページを検索する。
この段階では**タイトルと URL（または ID）のみ**を取得し、全文コンテンツは取得しない。

### 3-B-2. 候補をユーザーに提示して選択を求める

検索結果を番号付きリストで提示する。例：

```
以下のページが見つかりました。どれを取得しますか？

1. ログイン機能リファインメント結果 — https://www.notion.so/xxxxx
2. 認証フロー設計メモ — https://www.notion.so/yyyyy
3. OAuth 対応仕様 — https://www.notion.so/zzzzz

取得したいページの番号を教えてください。
```

**候補が 1 件に絞られた場合でも**、取得前に「このページで合っていますか？」と確認すること。

**候補が見つからなかった場合**は「該当するページが見つかりませんでした。別のキーワードで試しますか？」と聞き返す。

### 3-B-3. 選択されたページのみ全文を取得する

ユーザーが選んだページの ID を使い、Notion MCP でそのページのコンテンツを取得する。
**選ばれていないページの全文は取得しない。**

取得後はステップ 4（解釈・構造化）へ進む。

## ステップ 4 — コンテンツを解釈・構造化する

取得したコンテンツは形式が不統一であることを前提に解釈する。
（自由記述・箇条書き・データベースプロパティ・見出し構造など、様々な形式が混在する）

**取得対象がバックログアイテム（データベースページ）の場合**、データベースプロパティからバックログ ID を読み取り、解釈結果に含める。
バックログ ID が存在しない・取得できなかった場合は「バックログ ID が取得できませんでした。バックログ ID が設定されていない可能性があります。利用しているプロジェクト管理ツールの設定を確認してください。」とユーザーに伝える。その場合は解釈結果から省略し、それ以外の処理は通常通り続行する。
バックログアイテム以外のページ（仕様書など）を取得した場合はバックログ ID の読み取りは不要。

以下の 5 つのカテゴリに分類・解釈する：

| カテゴリ | 解釈の指針 |
|---|---|
| 背景 / Context | なぜこのタスクが必要か。課題・背景・きっかけ。 |
| ゴール / Goal | 実装方法ではなく、完了後に達成されるべき状態。 |
| 受け入れ基準 / Acceptance Criteria | 「完了」と判断できる具体的な条件。チェックボックス形式が望ましい。 |
| スコープ外 / Out of Scope | やらないこと、変更してはいけないもの。 |
| その他 | 上記に分類できなかった補足情報。 |

**解釈の進め方**：
- 内容から推察できる場合は、推察した内容をまとめてユーザーに提示し、
  「以上のように解釈しましたが、合っていますか？」と確認する。YES なら次へ進む。
- 推察できないフィールドがある場合はユーザーに直接ヒアリングする。
- 解釈に齟齬がある場合は修正して再確認する。

解釈の確認は、フィールドごとではなく**まとめて 1 回**行うのが望ましい。

## ステップ 5 — 構造化した内容を提示する

解釈・確認が完了したら、以下のフォーマットでコンソールに出力する。

---

**📋 Notion コンテンツ解釈結果**

**バックログ ID**: PBI-X
（バックログアイテムの場合のみ表示。取得できなかった場合は省略）

**背景 / Context**
<内容>

**ゴール / Goal**
<内容>

**受け入れ基準 / Acceptance Criteria**
<内容（チェックボックス形式が望ましい）>

**スコープ外 / Out of Scope**
<内容、または「特になし」>

**その他**
<分類できなかった情報があれば記載。なければ省略>

---

Markdown ファイルやその他のファイルへの書き出しは、ユーザーから明示的に指示があった場合のみ行う。

## 制約

- **この SKILL は取得と解釈のみを行う**。Issue の作成・コードベースの探索・ブランチ作成・
  ファイルへの自動書き出し・Notion への書き込みは、指示がない限り行わない。
- 解釈に不確かな点があれば作業を止めて確認する。推測で進めない。
- Notion MCP が使えない場合は代替手段（Notion API 直接呼び出し等）を提案せず、
  MCP の設定を促して止まる。
- 検索モードでは、ユーザーが選択したページのみ全文取得する。
  候補段階で全ページの全文を取得してコンテキストを浪費しない。
