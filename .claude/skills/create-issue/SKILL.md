# create-issue ワークフロー

このスキルは `.github/ISSUE_TEMPLATE/feature.yml` のフィールド構成に従って Issue の内容を組み立て、
`gh issue create` で GitHub に起票する。Issue の品質が Copilot Coding Agent の実装品質に直結するため、
各フィールドを丁寧に埋めることが目的。

## ステップ 1 — 起点を確認する

**引数ありで呼ばれた場合**（例：`/create-issue ログイン画面にOAuth対応を追加したい`）
→ その内容をヒアリングの出発点として使う。

**引数なしで呼ばれた場合**
→ 「何を実装・改善したいか、一言で教えてください。」と聞き返す。

また、**直前の会話スレッドに実装内容の議論がある場合**は、それを読み取って
「これまでのやり取りから〇〇という Issue を作成しようとしていると推察しますが、合っていますか？」と確認する。
YES なら ステップ 2 に進む。

**バックログとの連携**：直前のスレッドにバックログ ID（例: PBI-1）やバックログ URL（Notion URL）が含まれている場合は、それを取得して以降のステップで使用する。
含まれていない場合は「対応するバックログのバックログ ID またはバックログ URL があれば教えてください（なければスキップ可）」とヒアリングする。スキップされた場合はバックログ ID なし・バックログ URL なしとして以降を進める。

## ステップ 2 — フィールドをヒアリングする

各フィールドの扱いを以下のルールに従うこと。不明点があれば作業を止めて質問すること。

| フィールド | ルール |
|---|---|
| 背景 / Context | **必ずユーザーに聞き出す**。会話から推測できる場合でも、確認なしに決定しない。 |
| ゴール / Goal | **必ずユーザーに聞き出す**。会話から推測できる場合は「Goal は〇〇でよいですか？」と確認し、YES なら採用する。 |
| 受け入れ基準 / Acceptance Criteria | 「完了した」と判断できるチェックボックス形式の条件。会話から読み取れる場合は提案して確認する。 |
| スコープ外 / Out of Scope | やらないこと・変更してはいけないファイルや機能。「とくになし」のような回答も許容する。明示されると Copilot の暴走を防げることをユーザーに伝えると良い。 |

ヒアリングはまとめて 1 回で行うのが望ましい。確認が必要な項目を一度に提示し、
ユーザーが回答しやすいようにする。

## ステップ 3 — 実装ヒントを自動生成する

`Implementation Hints` はユーザーに聞かずに **コードベースを探索して自分で生成する**。

以下の観点でコードベースを調べ、Copilot が迷わないよう具体的に記載する：

- **変更対象ファイルの候補**：ゴールに関連するソースファイル・テストファイル・設定ファイル
- **類似実装の参照先**：既存の似た機能・パターンがあればそのファイルパスと概要
- **使うべきユーティリティ・インタフェース**：既存の共通処理、抽象クラス、ヘルパー関数など
- **注意すべき既存の制約**：既知の設計上の制約、命名規則、テストの書き方の慣習

**探索の進め方**（トークン消費を抑えるため、以下の順序で当たりをつけてから調査する）：

1. **ドキュメントを先に読む**：`README.md`・`CLAUDE.md`・`docs/` 以下のドキュメントがあれば最初に読み、
   アーキテクチャ・命名規則・ディレクトリ構成の概要を把握する。
2. **ディレクトリ構成を確認する**：`find . -maxdepth 3 -type f` などで構造を俯瞰し、
   関連しそうなディレクトリ・ファイルを絞り込む。
3. **絞り込んだ範囲だけ grep する**：コードベース全体への広範な grep は避け、
   ステップ 1・2 で特定した範囲に限定してキーワード検索する。

情報が少ない場合は「コードベースから推察できる範囲で記載しました」と補記する。

## ステップ 4 — 内容を提示してユーザーに最終確認する

Issue を作成する前に、組み立てた全フィールドをまとめてユーザーに提示し、
「この内容で Issue を作成しますか？」と確認する。修正があれば反映してから次へ進む。

バックログ ID がある場合、タイトルにバックログ ID プレフィックスが含まれていることを確認できるよう提示する。例：
> **タイトル**: PBI-1 ログイン画面にOAuth対応を追加する

## ステップ 5 — Issue を作成する

確認が取れたら `gh issue create` で Issue を起票する。

**タイトルの形式**：
- バックログ ID がある場合：`PBI-X タイトル` 形式にする（例: `PBI-1 ログイン画面にOAuth対応を追加する`）
- バックログ ID がない場合：プレフィックスなしのタイトルをそのまま使う

```bash
gh issue create \
  --title "<タイトル>" \
  --label "feature" \
  --body "$(cat <<'EOF'
## 背景 / Context

<context の内容>

## ゴール / Goal

<goal の内容>

## 受け入れ基準 / Acceptance Criteria

<acceptance_criteria の内容（チェックボックス形式）>

## 実装ヒント / Implementation Hints

<implementation_hints の内容>

## スコープ外 / Out of Scope

<out_of_scope の内容>

## バックログ / Backlog

<バックログの Notion URL。バックログ ID もバックログ URL もない場合は空欄のまま>
EOF
)"
```

作成後は **Issue の URL をユーザーに提示する**。

## 制約

- **Issue 作成以外の操作は行わない**。ブランチ作成・PR 作成・Assignee の変更・Milestone の設定など、
  依頼されていない操作は実行しない。ラベル `feature` の付与は上記コマンドに含まれているため許容。
- ヒアリング中に不明点が出たら、推測で進めずに質問する。
- `gh` コマンドが使えない環境では、その旨をユーザーに伝えて内容だけ提示する。
