# TDD ワークフロー（Red → Green → Refactoring）

このプロジェクトでは Kent Beck / 和田卓人スタイルのテスト駆動開発で、ガードレールを敷きつつステップバイステップに開発を進めます。この skill が起動した時点から、以下の 4 ステップに従って作業してください。

## 最重要制約: ステップごとに止めてレビュー依頼する

**この skill の核となる制約です。** 4 つのステップそれぞれの完了時に、必ず作業を止めてユーザーにレビューを依頼してください。各ステップで何をレビューしてほしいのかは後述しますが、ユーザーが各段階で別々の観点で確認したいために細かく刻んでいる、という意図を踏まえて差分を提示してください。

ただし、修正の内容が明らかに簡単で、レビューの負担も少ないと判断できる場合に限り、自分の判断で次のステップまで進めて構いません。その場合は **「このステップは軽微なのでまとめて N まで進めました」と必ず明示**してから、続きをユーザーに見せてください。判断に迷うときは止める方を選んでください。

不明点・前提が確認できない点があれば、勝手に決めずに作業を止めて質問してください。

## 引数の扱い

- `/tdd <要件説明>` のように引数で実装したい機能の説明が渡された場合は、それを最初の TDD サイクルの題材として扱ってください。
- 引数なしで起動された、または会話の流れで自然に発動した場合は、まず「これから TDD で進めます。何を実装したいか教えてください」と尋ねてから始めてください。

## 開発ステップ

### 1. [RED] テストコードのみを作成・修正する

- **テストコードのみ**を書きます。プロダクションコードはまだ書きません。
- 対象のプロダクションコードがまだ存在しない、もしくはシグネチャが揃っていないことによるコンパイルエラー（または相当するエラー）が発生することは許容します。
- ここで止めてレビュー依頼。**観点はテストの書き方・仕様の充足度・仕様理解・プロダクションコードを書く前のガードレールとしての正しさ**です。

### 2. [RED → GREEN] コンパイルエラーを解消させる「必要最小限」の実装を行う

- テストを通すための実装ではなく、**コンパイル（または型チェック・解決）が通る最小限**のスケルトン実装に留めてください。返り値は仮の値で構いません。
- ここで止めてレビュー依頼。**観点はプロダクションコードの実装方針に誤りがないか**です。設計の方向性をここで合意します。

### 3. [GREEN] テストが通る「必要最低限」の実装を行う

- ステップ 1 で書いたテストを通すための、**必要最低限**の実装を行います。
- まだリファクタリングはしません。多少の重複・不格好さは残して構いません。
- ここで止めてレビュー依頼。**観点はテストコード（仕様）のガードレールに則った具体的な実装の方針**です。

### 4. [REFACTORING] テストが通る状態を維持しながらリファクタリングを実施する

- すべてのテストが通る状態を維持したまま、重複の除去・命名の改善・責務の分離などを行います。
- ここで止めてレビュー依頼。**観点は実装の最終結果**です。

## 共通の指針

### テストコードの記法・命名規則

対象プロジェクトに既存のテストコードがあれば、その記法・命名規則を踏襲してください。ただし、既存テストの記法・命名規則に統一感がない、もしくはそもそも既存のテストコードがない場合は、**参考にすべきテストコードの記法・命名規則をユーザーに確認**してから書き始めてください。

### モック・テストダブルの扱い

テスト対象に直接関係のない依存先はテストダブルに差し替えてテストしてください。差し替えの手法（モックライブラリ・スタブ・フェイクなど）については、**既存プロジェクトのテストでよく使われているモック手法があればそれを踏襲**してください。既存に明確なパターンがない場合はユーザーに確認してください。

### 質問

何かわからないこと、前提が立てられないこと、複数の選択肢があって判断に迷うことがあれば、想像で進めずに作業を止めて質問してください。
