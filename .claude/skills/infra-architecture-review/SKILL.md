# infra-architecture-review: クラウドインフラ構成のシニアアーキテクト・レビュー

## このスキルの目的

既存のクラウドインフラ構成（Terraform / CDK / Pulumi / CloudFormation / Kubernetes / 構成図など）をレビューし、**「クラウドインフラのシニアアーキテクトが見て納得できる水準か」** を評価したレポートを出力します。

このスキルは改善 TODO を作りません。**評価レポートのみ**を出力します。TODO 化・実装計画は別スキル「infra-improvement-todo」の責務なので踏み込まないでください。

> このスキルは `SKILL.md` 単体で自己完結します。外部 URL に依存しません（任意で参考 URL を取得してもよいが、無くても本文の埋め込み知識だけで評価を完結できること）。Claude 以外のエージェントでも、このファイルだけ読めば同じ評価が再現できるように記述しています。

## いつ発動するか（トリガー）

- 「インフラ / IaC をレビューして / 評価して」「ベストプラクティスに沿っているか見て」
- 「アーキテクチャを監査して」「この構成どう思う？」
- クラウド構成（AWS / GCP / Azure を問わず）の品質評価依頼全般

## 最重要の評価思想（ここを外すとこのスキルの意味がない）

Well-Architected Framework（AWS / Google Cloud / Azure の 3 社がいずれも提唱）の本質は、**「ビジネスコンテキストに紐づくトレードオフを意識的に下す」ための問いとリスクのフレームワーク**であって、「全ベストプラクティスを 100% 実装する」チェックリストではありません（3 社とも "It depends" を明言しています）。

したがって、このスキルでは次の **2 点だけ** で評価します。

1. **アンチパターン**（各社が明確に「避けよ」と言う設計）を踏んでいないか
2. **各トレードオフを業務優先度（= 後述の SLO）で正当化できるか**

ベストプラクティス未充足でも、業務優先度に照らして正当化できれば **許容** とします。「指摘＝要修正」ではありません。「正当なトレードオフ（直す必要なし）」を結論として出すことを明示的に許可します。

## ガードレール（絶対に守る）

- Well-Architected を「100% 達成すべきチェックリスト」として扱わない。**アンチパターン回避 + SLO での正当化**の 2 点で評価する。
- トレードオフの正当化は必ず **SLO（定量化した業務優先度）** に紐づける。「一般論」だけを根拠にしない。
- **アプリケーションアーキテクチャを所与の正解とみなさない。** 業務（SLO）に対する松竹梅ミスマッチを検出したら、インフラを合わせるのではなく**アプリ側の見直しを示唆**する。
- 重要度（高 / 中 / 低）を唯一の分類軸にしない。**過剰設計も過少設計と同等に**指摘する。
- 「正当なトレードオフ（＝直す必要なし）」を結論として許可する。
- 特定ベンダ語彙に固執せず 3 社共通の思想で評価する。ただし対象が特定ベンダなら、根拠は**具体的なサービス名**で示す。
- このスキルは**評価のみ**。TODO 化・実装計画には踏み込まない。

## 評価の背骨：Well-Architected の 6 つの柱（ベンダ非依存に抽象化）

各柱を必ず一度は当て、「該当あり / 問題なし / 対象外」を記録してください（網羅性の担保）。

1. **運用上の優秀性** — IaC 化、デプロイ自動化、可観測性（ログ / メトリクス / トレース）、変更の安全性
2. **セキュリティ** — 最小権限、多層防御、保存時 / 転送時暗号化、シークレット管理、監査証跡、攻撃面の最小化
3. **信頼性** — 冗長化、障害分離、自動復旧、バックアップ / 復旧（RPO / RTO）、非同期処理の失敗ハンドリング（リトライ・DLQ・冪等性）、スケーラビリティ
4. **パフォーマンス効率** — 適切なリソース選定、読み書き分離、キャッシュ、ボトルネック特定、需要追従
5. **コスト最適化** — 過剰プロビジョニング回避、需要追従、マネージド活用、環境別の投資メリハリ（dev で絞る判断は減点ではなく**加点**）
6. **持続可能性** — リソース効率、不要リソース削減、適切なサイジング

## 手順 0: ビジネスコンテキストを SLO で定量化する（評価の前提・最初に必ず実施）

トレードオフの正当化は「一般論」ではなく、**定量化された業務優先度 = SLO** に紐づけて初めて成立します。だから評価の前に、まず SLO をユーザにヒアリングします。

ただし多くのプロダクトは SLO が未定義です。**白紙で「SLO を教えてください」と聞かないこと。** 検出した構成階層（後述の「松竹梅」）に応じた**候補 SLI / SLO を提案して、ユーザに反応してもらう**のが正しいやり方です。ユーザが提供できない場合は「想定 SLO」を明示的に置き、推測である旨を記録します。

### 用語の階層（混同しない）

- **SLI** = 測定値（測る対象）
- **SLO** = 内部目標（例: 可用性 99.9%、計測窓は rolling 28 日など）
- **SLA** = 顧客契約上の約束（違反に責任が伴う）
- **エラーバジェット** = (100% − SLO) として許容される逸脱量

### SLI の代表種別（ベンダ非依存。Google SRE / New Relic / PagerDuty 等が共通して挙げるもの）

- 可用性（成功リクエスト率 / 稼働率）
- レイテンシ（応答時間 p50 / p95 / p99）
- エラー率
- スループット / トラフィック
- 飽和度（リソース使用率）
- 耐久性（データが消失しないこと。ストレージ）
- 鮮度 / 正確性（非同期処理・データパイプラインの遅延や整合。例: CQRS / ES の read model 反映遅延、DLQ 滞留、投影ラグ）

### 階層に応じて必要な SLI が変わる（提案の指針）

- **梅**: 可用性 + レイテンシ程度で十分
- **竹**: + エラー率 / 飽和度 / 復旧目標（RPO / RTO）
- **松**: + サービス別 SLO / イベント投影の鮮度 / DLQ 滞留 / 障害ドメイン別の独立性

> 参考（任意。WebFetch 可。無くても上記で評価できる）: Google SRE Workbook、New Relic / PagerDuty の SLO/SLI 解説。**このスキルはこれらの URL に依存せず、上記の埋め込み知識で自己完結すること。**

## 「松竹梅」モデル：構成階層が課題ドメインに見合っているかを評価する

アプリケーションアーキテクチャの 3 つの典型です（優劣ではなく**適材適所**）。

- **梅**: MVC + トランザクションスクリプト + モノリス
- **竹**: クリーンアーキテクチャ（ヘキサゴナル / オニオン）+ DDD + モジュラモノリス
- **松**: CQRS & イベントソーシング + アクターモデル & メッセージ駆動 + マイクロサービス

対応するインフラ構成の典型例（AWS を例にした参考。GCP / Azure は同型に読み替えること）:

| 観点 | 梅: 三層モノリス | 竹: コンテナ + Multi-AZ | 松: イベント駆動マイクロサービス |
|---|---|---|---|
| 対応アプリ層 | MVC + Tx Script + モノリス | Clean Arch + DDD + モジュラモノリス | CQRS+ES + アクター + マイクロサービス |
| コンピュート | 単一 / 小規模オートスケール | コンテナ (2 AZ) | コンテナ + FaaS（多サービス） |
| データ | RDB Single-AZ | RDB Multi-AZ + キャッシュ | イベントストア + リードモデル分離 |
| 結合 | 同期・密結合 | 同期中心・モジュール境界 | 非同期・メッセージング |
| 運用 | 手動デプロイ・最小ログ | IaC + CI/CD + トレース | GitOps + 統一可観測性 |
| セキュリティ | FW + 最小限 | WAF + シークレット管理 + 最小権限 | 多層防御 + アカウント分離 + mTLS |
| 信頼性 | 単一障害点あり | Multi-AZ 冗長 | 障害ドメイン分離 + Saga + DLQ |
| パフォーマンス | 垂直スケール | 水平オートスケール | サービス別最適化 + 読み書き分離 |
| コスト | 最小 | 中 | 高（運用人件費含め最大） |
| 適する条件 | PoC・小規模・短命 | 大半のプロダクト（最良デフォルト） | 高複雑性ドメイン・大規模・高スループット |

### 評価上の使い方（重要：アプリを正解の前提にしない）

1. アプリ層が松竹梅のどれかを推定し、**インフラ階層が見合っているか**を見る。
2. ミスマッチを検出したら、**修正の向きを診断する**。アプリを所与とせず、次の 2 方向を検討する:
   1. **インフラが過剰 / 過少**（アプリ階層に対して）→ インフラ側の指摘
   2. **アプリ階層が業務（SLO）に対して過剰 / 過少** → この場合、インフラを合わせると過剰 / 過少を増幅するだけ。**アプリ側アーキテクチャの見直しを示唆**する（例: PoC / 小規模なのに松を採用＝オーバーエンジニアリング、SLO が高いのに梅＝信頼性不足）。
3. **「竹が大半のプロダクトの最良デフォルト」を基準線**に、過剰も過少も等しく減点する。

## 評価ワークフロー

1. **SLO で業務コンテキストを定量化**（手順 0）。階層に応じた候補 SLI / SLO を提案してから確定する。
2. **構成の棚卸し**。単一成果物の lint で終わらせない。**実際の利用者（アプリコード / CI・CD / デプロイマニフェスト / 実デプロイ状態）と突き合わせ**、「設定したが使われていない / 有効だが未配線」という整合性破れを検出する。
3. **松竹梅の見立て + 向きの診断**（上記）。
4. **6 柱でカバレッジ**。各柱でアンチパターンの有無と、各トレードオフの SLO 整合性を評価する。
5. **総合判定**。シニアアーキテクトが納得する水準かを言語化する。

---

## 各指摘の分類軸（重要度だけで分類しない）

各指摘には、以下の 4 軸のタグを付けてください。重要度（高 / 中 / 低）を唯一の軸にしないこと。

- **状態**: 壊れている（意図通り動かない） / 最適でない（動くが改善余地あり）
- **性質**: 設計・本質（環境非依存） / ダイヤル（環境依存パラメータ）
- **正当性**: 正当化不能（SLO で説明できない） / 正当なトレードオフ（SLO で説明可能）
- **帰属**: インフラ側 / アプリ側（向きの診断結果）

## 出力フォーマット（評価レポート 1 本。TODO は作らない）

以下の見出し構成で、**日本語**のレポートを 1 本出力してください。

1. **総評** — シニアアーキテクトが納得する水準かの結論
2. **前提（ビジネスコンテキスト / SLO）** — ヒアリング結果。未定義なら、提案した候補と置いた想定値を明示（推測である旨を記録）
3. **松竹梅の見立て** — アプリ層の階層推定、インフラとの整合 / ミスマッチ、ミスマッチの**向き診断**
4. **6 柱別の評価** — 各柱ごとに「良い点 / アンチパターン / SLO で正当化できないトレードオフ」。各指摘に**分類軸タグ**と**根拠**（`file:line`・層をまたいだ証拠）を付ける
5. **アプリケーション側への示唆**（out-of-scope だが重要） — インフラとの整合や SLO 整合の観点でアプリ側を直すべきと判断した点。「インフラを合わせるのではなくアプリを直す」候補
6. **整合している点（良い設計）** — 後続作業で壊さないための記録
7. **判定** — 「シニアが納得する天井」に対し、現状がどこにあるか

## 良い例 / 悪い例

**悪い例（やってはいけない）:**
- SLO を聞かず、一般論でベストプラクティス未充足を片端から「課題」化する
- アプリを正解と決めつけ、インフラだけを見る
- 過剰設計（オーバーエンジニアリング）を見逃す
- 重要度ラベルだけを付けて終わる

**良い例（こうあるべき）:**
- SLO を提案・確定し、各トレードオフを SLO で正当化する
- 松竹梅ミスマッチの**向き**を診断し、必要ならアプリ側修正を示唆する
- 6 柱で網羅する
- シニアが納得する水準かを明言する

## スコープ
