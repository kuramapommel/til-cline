# infra-architecture-review: クラウドインフラ構成のシニアアーキテクト・レビュー

## このスキルの目的

既存のクラウドインフラ構成（Terraform / CDK / Pulumi / CloudFormation / Kubernetes / 構成図など）をレビューし、**「クラウドインフラのシニアアーキテクトが見て納得できる水準か」** を評価したレポートを出力します。

このスキルは改善 TODO を作りません。**評価レポートのみ**を出力します。TODO 化・実装計画は別スキル「infra-improvement-todo」の責務なので踏み込まないでください。

> このスキルは `SKILL.md` 単体で自己完結します。外部 URL に依存しません（任意で参考 URL を取得してもよいが、無くても本文の埋め込み知識だけで評価を完結できること）。Claude 以外のエージェントでも、このファイルだけ読めば同じ評価が再現できるように記述しています。

## いつ発動するか（トリガー）

- 「インフラ / IaC をレビューして / 評価して」「ベストプラクティスに沿っているか見て」
- 「アーキテクチャを監査して」「この構成どう思う？」
- クラウド構成（AWS / GCP / Azure を問わず）の品質評価依頼全般

## インフラ設計の基本フロー（レビューはこの「逆引き検証」である）

健全なインフラ設計は、次の流れで「業務要求 → 指標 → 目標 → 設計 → 継続改善」と落ちていきます。

```
① 非機能要求グレードで要件定義
   （可用性99.9%・応答3秒以内 など、業務が求める非機能要求を洗い出す）
        ↓
② 各要件に SLI を選定
   （可用性→成功リクエスト率 ／ 性能→p95レイテンシ）
        ↓
③ SLI ごとに SLO/SLA を設定
   （成功率≥99.9% ／ p95<3,000ms）
        ↓
④ SLO ドリブンに Well-Architected の柱へ落として設計
   （99.9%→信頼性: Multi-AZ, AutoScaling ／ 3秒→性能効率: CDN, キャッシュ）
        ↓
⑤ 監視 + W-A Review で継続見直し
   （SLO達成→エラーバジェット管理 ／ W-A Review→改善点特定→③④へフィードバック）
```

**このスキルがやるレビューは、この設計フローの逆引き検証です。** 出来上がったインフラ（④の成果物）を見て、それが ③ の SLO/SLA を構造的に満たせるか、② の SLI にヌケモレがないか、そして ① の要件網羅に穴がないかを遡って確かめます。だから評価ワークフローも「SLI のヌケモレ（②）→ SLO/SLA の確認（③）→ SLO ドリブン評価（④の妥当性）→ アンチパターン評価（④の質）→ エラーバジェット所見（⑤の入口）」という順で進みます。

## 最重要の評価思想（ここを外すとこのスキルの意味がない）

Well-Architected Framework（AWS / Google Cloud / Azure の 3 社がいずれも提唱）の本質は、**「ビジネスコンテキストに紐づくトレードオフを意識的に下す」ための問いとリスクのフレームワーク**であって、「全ベストプラクティスを 100% 実装する」チェックリストではありません（3 社とも "It depends" を明言しています）。

したがって、このスキルでは次の **2 点だけ** で評価します。

1. **アンチパターン**（各社が明確に「避けよ」と言う設計）を踏んでいないか
2. **各トレードオフを業務優先度（= SLO）で正当化できるか**

ベストプラクティス未充足でも、業務優先度に照らして正当化できれば **許容** とします。「指摘＝要修正」ではありません。「正当なトレードオフ（直す必要なし）」を結論として出すことを明示的に許可します。

## ガードレール（絶対に守る）

- Well-Architected を「100% 達成すべきチェックリスト」として扱わない。**アンチパターン回避 + SLO での正当化**の 2 点で評価する。
- トレードオフの正当化は必ず **SLO（定量化した業務優先度）** に紐づける。「一般論」だけを根拠にしない。
- **アプリケーションアーキテクチャを所与の正解とみなさない。** 業務（SLO）に対する松竹梅ミスマッチを検出したら、インフラを合わせるのではなく**アプリ側の見直しを示唆**する。
- 重要度（高 / 中 / 低）を唯一の分類軸にしない。**過剰設計も過少設計と同等に**指摘する。
- 「正当なトレードオフ（＝直す必要なし）」を結論として許可する。
- 特定ベンダ語彙に固執せず 3 社共通の思想で評価する。ただし対象が特定ベンダなら、根拠は**具体的なサービス名**で示す。
- **IPA 非機能要求グレードは「要件ヌケモレ検出の補助チェックリスト」**として使う。評価の背骨はあくまで Well-Architected のベンダ非依存思想であり、日本ローカル文脈に寄せすぎない。
- **「全チェック項目を漏れなく逐次評価」を強制しない。**「全柱を漏れなく当てる」+「問題なしは簡潔に」を守る（"It depends" の堅持 = トークン節約とも整合）。
- **エラーバジェットは、設計レビュー時点では「設計が構造的に SLO/SLA を満たせるか（構造的な黒字 / 赤字）」を指す。** 運用実測のバジェット消費とは区別する。
- **SLI / SLO / SLA のヒアリングは 1 メッセージに集約**し、往復を最小化する。未確定なら想定値でフォールバックして評価を止めない。
- このスキルは**評価のみ**。TODO 化・実装計画には踏み込まない。

## 評価の背骨：Well-Architected の 6 つの柱（ベンダ非依存に抽象化）

1. **運用上の優秀性** — IaC 化、デプロイ自動化、可観測性（ログ / メトリクス / トレース）、変更の安全性
2. **セキュリティ** — 最小権限、多層防御、保存時 / 転送時暗号化、シークレット管理、監査証跡、攻撃面の最小化
3. **信頼性** — 冗長化、障害分離、自動復旧、バックアップ / 復旧（RPO / RTO）、非同期処理の失敗ハンドリング（リトライ・DLQ・冪等性）、スケーラビリティ
4. **パフォーマンス効率** — 適切なリソース選定、読み書き分離、キャッシュ、ボトルネック特定、需要追従
5. **コスト最適化** — 過剰プロビジョニング回避、需要追従、マネージド活用、環境別の投資メリハリ（dev で絞る判断は減点ではなく**加点**）
6. **持続可能性** — リソース効率、不要リソース削減、適切なサイジング

> AWS は 6 柱。Google Cloud / Azure は持続可能性を独立柱に立てず 5 柱とする場合があるが、思想は同型。ベンダによらず上記 6 観点で当てる。

## IPA 非機能要求グレード × Well-Architected 対応マッピング（二重チェックを避けるための 1 枚）

IPA 非機能要求グレードの 6 大項目と W-A の 6 柱は**重複が大きい**。同じ観点を IPA 名・W-A 名で 2 度 3 度確認するのはトークンの無駄でユーザビリティも悪い。そこで両者を 1 枚にマッピングし、**右端の「統合観点」を 1 回だけ走らせる**。IPA 側にしかない観点（移行性）、W-A 側で強調される観点（コスト最適化・持続可能性）も、この表で取りこぼさない。

| IPA 非機能要求グレード（6 大項目） | 対応する W-A 柱 | 統合観点（これを 1 回だけ評価する） |
|---|---|---|
| 可用性 | 信頼性 | **① 可用性・信頼性** |
| 性能・拡張性 | パフォーマンス効率（+ 信頼性のスケーラビリティ） | **② 性能・拡張性** |
| 運用・保守性 | 運用上の優秀性 | **③ 運用・保守性** |
| セキュリティ | セキュリティ | **④ セキュリティ** |
| （IPA では性能・運用に分散） | コスト最適化 | **⑤ コスト最適化**（W-A 側で強調） |
| システム環境・エコロジー | 持続可能性 | **⑥ 持続可能性・エコロジー** |
| 移行性 | （W-A に独立柱なし。運用上の優秀性に一部） | **⑦ 移行性**（IPA 固有）: データ / システム移行方式、切替・並行稼働、ロールバック |

> 評価では、IPA 項目名・W-A 柱名を別々に往復せず、**右端の統合観点 ①〜⑦ を 1 周**する。①〜⑥ は上の W-A 6 柱と 1:1 対応なので、各観点の評価内容は 6 柱リストの記述を参照する（ここで再列挙しない＝二重記述の回避）。レポートの「6 柱別の評価」はこの ①〜⑥ に対応させ、⑦ 移行性は W-A に独立柱がないため必要時に補助観点として加える。

## 用語の階層（混同しない）

- **SLI** = 測定値（測る対象）
- **SLO** = 内部目標（例: 可用性 99.9%、計測窓は rolling 28 日など）
- **SLA** = 顧客契約上の約束（違反に責任が伴う）
- **エラーバジェット** = (100% − SLO) として許容される逸脱量。**設計レビューでは「構造的な黒字 / 赤字」**を指す（後述）。

### SLI の代表種別（ベンダ非依存。Google SRE / New Relic / PagerDuty 等が共通して挙げるもの）

- 可用性（成功リクエスト率 / 稼働率）
- レイテンシ（応答時間 p50 / p95 / p99）
- エラー率
- スループット / トラフィック
- 飽和度（リソース使用率）
- 耐久性（データが消失しないこと。ストレージ）
- 鮮度 / 正確性（非同期処理・データパイプラインの遅延や整合。例: CQRS / ES の read model 反映遅延、DLQ 滞留、投影ラグ）

> 参考（任意。WebFetch 可。無くても上記で評価できる）: Google SRE Workbook、New Relic / PagerDuty の SLO/SLI 解説、IPA 非機能要求グレード。**このスキルはこれらの URL に依存せず、上記の埋め込み知識で自己完結すること。**

## 「松竹梅」モデル：構成階層が課題ドメインに見合っているかを評価する

アプリケーションアーキテクチャの 3 つの典型です（優劣ではなく**適材適所**）。

- **梅**: MVC + トランザクションスクリプト + モノリス
- **竹**: クリーンアーキテクチャ（ヘキサゴナル / オニオン）+ DDD + モジュラモノリス
- **松**: CQRS & イベントソーシング + アクターモデル & メッセージ駆動 + マイクロサービス

対応するインフラ構成の典型例（AWS を例にした参考。GCP / Azure は同型に読み替えること）:

| 観点 | 梅: 三層モノリス | 竹: コンテナ + Multi-AZ | 松: イベント駆動マイクロサービス |
|---|---|---|---|
| 対応アプリ層 | MVC + Tx Script + モノリス | Clean Arch + DDD + モジュラモノリス | CQRS+ES + アクター + マイクロサービス |
| コンピュート | 単一 / 小規模オートスケール | コンテナ (2 AZ) | コンテナ + FaaS（多サービス） |
| データ | RDB Single-AZ | RDB Multi-AZ + キャッシュ | イベントストア + リードモデル分離 |
| 結合 | 同期・密結合 | 同期中心・モジュール境界 | 非同期・メッセージング |
| 運用 | 手動デプロイ・最小ログ | IaC + CI/CD + トレース | GitOps + 統一可観測性 |
| セキュリティ | FW + 最小限 | WAF + シークレット管理 + 最小権限 | 多層防御 + アカウント分離 + mTLS |
| 信頼性 | 単一障害点あり | Multi-AZ 冗長 | 障害ドメイン分離 + Saga + DLQ |
| パフォーマンス | 垂直スケール | 水平オートスケール | サービス別最適化 + 読み書き分離 |
| コスト | 最小 | 中 | 高（運用人件費含め最大） |
| 適する条件 | PoC・小規模・短命 | 大半のプロダクト（最良デフォルト） | 高複雑性ドメイン・大規模・高スループット |

### 階層に応じて必要な SLI が変わる（提案の指針）

- **梅**: 可用性 + レイテンシ程度で十分
- **竹**: + エラー率 / 飽和度 / 復旧目標（RPO / RTO）
- **松**: + サービス別 SLO / イベント投影の鮮度 / DLQ 滞留 / 障害ドメイン別の独立性

### 向き診断（重要：アプリを正解の前提にしない）

1. アプリ層が松竹梅のどれかを推定し、**インフラ階層が見合っているか**を見る。
2. ミスマッチを検出したら、**修正の向きを診断する**。アプリを所与とせず、次の 2 方向を検討する:
   1. **インフラが過剰 / 過少**（アプリ階層に対して）→ インフラ側の指摘
   2. **アプリ階層が業務（SLO）に対して過剰 / 過少** → この場合、インフラを合わせると過剰 / 過少を増幅するだけ。**アプリ側アーキテクチャの見直しを示唆**する（例: PoC / 小規模なのに松を採用＝オーバーエンジニアリング、SLO が高いのに梅＝信頼性不足）。
3. **「竹が大半のプロダクトの最良デフォルト」を基準線**に、過剰も過少も等しく指摘する。

## 評価ワークフロー（手順 1 〜 6）

設計フローの逆引きとして、次の 6 手順で進める。

### 手順 1: SLI のヌケモレ評価（要件の網羅性チェック）

設計フローの ① 要件定義 → ② SLI 選定に穴がないかを確かめる。**「IPA×W-A 対応マッピング」の統合観点 ①〜⑦ を 1 回だけ**走らせ、各観点について「この構成で測るべき SLI が用意されているか」を確認する。

- 統合観点リストを使うことで、IPA 名・W-A 名での**二重チェックを避ける**（同じ観点を 2 度 3 度確認しない）。
- 評価の背骨は W-A のベンダ非依存思想。IPA は「日本でよく使われる要件ヌケモレ検出のチェックリスト」として補助的に当てる（移行性・エコロジーなど W-A の柱名に現れにくい観点の取りこぼし防止）。
- 検出した松竹梅階層に応じて、**必要 SLI の候補を提案**する（白紙で「SLI を教えて」と聞かない）。

### 手順 2: SLI のヌケモレを通知

欠けている指標があれば、**具体的にどの統合観点のどの SLI が欠けているか**をユーザに知らせ、「後ほど改めて SLI を定義する」ことを提案する。ここで評価を止めない（即時のフォールバックは手順 3 で行う）。

### 手順 3: SLO/SLA のヒアリング（往復を最小化）

各 SLI について、SLO/SLA をどの値で設定・合意しているかをヒアリングする。

- **手順 1 の候補 SLI と、推奨 SLO/SLA レンジを 1 メッセージにまとめて提示し、一括で確認する。** SLI 用・SLO 用で 2 回別々に往復しない（UX 重視）。
- ユーザが答えられない / 今すぐ評価したい場合は、**想定 SLO/SLA を明示的に置き、推測である旨を記録**してフォールバックする。評価を止めない。

### 手順 4: ［評価 1］SLA/SLO ドリブン評価

設計（フローの ④）が、合意された（または想定した）SLO/SLA を**構造的に満たせるか**を判定する。

- **SLA を満たせない（または設計上満たすのが困難と判断できる）箇所** → 対象箇所と「どの SLI をどう満たせていないか」を **修正必須レベル** でアラートする。
- **SLO を満たせない（または設計上困難）箇所** → 同様の情報を **注意・提案レベル** で通知する。
- これは 4 分類軸の「正当性」と接続する: **SLA 未達 = 正当化不能（修正必須）**、**SLO 未達 = 要トレードオフ判断（注意・提案）**。

### 手順 5: ［評価 2］Well-Architected ベストプラクティス / アンチパターン評価

6 本の柱それぞれについて、**アンチパターンに該当しうる観点を中心に**対象インフラを評価する。

- **「全チェック項目を漏れなく逐次評価」ではなく「全柱を漏れなく当てる」。** 各柱で「該当あり / 問題なし / 対象外」を記録し、**問題なしは簡潔に**。100% チェックリスト化しない（"It depends"）を堅持し、手順 1 のトークン節約意図と整合させる。
- ここで**松竹梅の向き診断**を行う。アプリ階層に対しインフラが過剰 / 過少か、あるいはアプリ階層が業務（SLO）に対し過剰 / 過少か。後者ならアプリ側修正を示唆する。竹を最良デフォルトの基準線とし、過剰も過少も等しく指摘する。

> 構成の棚卸しは単一成果物の lint で終わらせない。**実際の利用者（アプリコード / CI・CD / デプロイマニフェスト / 実デプロイ状態）と突き合わせ**、「設定したが使われていない / 有効だが未配線」という整合性破れを検出する。

### 手順 6: エラーバジェットでアウトプットのトーンを調整

**設計レビュー時点では運用実測がない**のが普通。よってここで言うエラーバジェットは「**設計が構造的に SLO/SLA を満たせるか（構造的な黒字 / 赤字）**」を指す。運用実測のバジェット消費とは区別する。

このスキルは TODO を作らない（責務外）。調整するのは**アウトプットのトーン（行動の強制度）だけ**。

- **十分に黒字（構造的に余裕あり）**: 次回以降の改善優先順は示すが、評価結果の照会にとどめ、次の行動は促さない。
- **残りわずか（黒字だが薄い / 構造的に際どい）**: 使い切りが近い旨を強めに注意喚起するが、行動は強制しない。
- **赤字（設計上 SLA/SLO を満たせない箇所がある）**: 修正必須。改善優先順の高いものから順に、ユーザに行動を**優しく**促す。

## 各指摘の分類軸（重要度だけで分類しない）

各指摘には、以下の 4 軸のタグを付けてください。重要度（高 / 中 / 低）を唯一の軸にしないこと。

- **状態**: 壊れている（意図通り動かない） / 最適でない（動くが改善余地あり）
- **性質**: 設計・本質（環境非依存） / ダイヤル（環境依存パラメータ）
- **正当性**: 正当化不能（SLO で説明できない） / 正当なトレードオフ（SLO で説明可能）。手順 4 と接続: SLA 未達 = 正当化不能 = 修正必須 / SLO 未達 = 要トレードオフ判断 = 注意・提案
- **帰属**: インフラ側 / アプリ側（向きの診断結果）

## 出力フォーマット（評価レポート 1 本。TODO は作らない）

以下の見出し構成で、**日本語**のレポートを 1 本出力してください。

1. **総評** — シニアアーキテクトが納得する水準かの結論
2. **前提（ビジネスコンテキスト / SLO）** — 次を含める:
   - **SLI ヌケモレ**（手順 1〜2）: IPA×W-A 統合ビュー（観点 ①〜⑦）で、どの観点のどの SLI が欠けているか
   - **SLO/SLA 合意値（または想定値）**（手順 3）: 合意済みの値、または明示的に置いた想定値（推測である旨を記録）
3. **松竹梅の見立て** — アプリ層の階層推定、インフラとの整合 / ミスマッチ、ミスマッチの**向き診断**
4. **6 柱別の評価** — 各柱（統合観点 ①〜⑥。⑦ 移行性は必要時に補助）ごとに、次の双方を織り込む:
   - **［評価 1］SLA/SLO 達成可否**（手順 4）: SLA 未達 = **修正必須** / SLO 未達 = **注意・提案** のレベルを明示
   - **［評価 2］アンチパターン**（手順 5）: 良い点 / アンチパターン / SLO で正当化できないトレードオフ
   - 各指摘に**分類軸タグ**と**根拠**（`file:line`・層をまたいだ証拠）を付ける。問題なしの柱は簡潔に
5. **アプリケーション側への示唆**（out-of-scope だが重要） — インフラとの整合や SLO 整合の観点でアプリ側を直すべきと判断した点。「インフラを合わせるのではなくアプリを直す」候補
6. **整合している点（良い設計）** — 後続作業で壊さないための記録
7. **判定** — 「シニアが納得する天井」に対し、現状がどこにあるか
8. **エラーバジェット所見**（手順 6） — 設計が**構造的に黒字 / 赤字**か（運用実測ではない旨を明記）と、それに応じた**トーン**（黒字 = 照会のみ / 残りわずか = 注意喚起 / 赤字 = 修正必須を優しく促す）

## 良い例 / 悪い例

**悪い例（やってはいけない）:**
- SLO を聞かず、一般論でベストプラクティス未充足を片端から「課題」化する
- IPA 項目名と W-A 柱名で同じ観点を 2 度 3 度チェックする（統合観点リストを使わない）
- 全チェック項目を逐次 100% 評価し、問題なしの柱まで冗長に書く
- アプリを正解と決めつけ、インフラだけを見る
- 過剰設計（オーバーエンジニアリング）を見逃す
- エラーバジェットを運用実測の話と混同する
- SLI 用・SLO 用で別々に往復してユーザを疲れさせる
- 重要度ラベルだけを付けて終わる

**良い例（こうあるべき）:**
- 統合観点 ①〜⑦ を 1 回走らせて SLI ヌケモレを検出し、候補 SLI と推奨 SLO/SLA を 1 メッセージで提示する
- SLA 未達 = 修正必須 / SLO 未達 = 注意・提案 を明確に分けてアラートする
- 6 柱を漏れなく当てつつ、問題なしは簡潔に書く
- 松竹梅ミスマッチの**向き**を診断し、必要ならアプリ側修正を示唆する
- 末尾で構造的な黒字 / 赤字を述べ、それに応じたトーンで締める
- シニアが納得する水準かを明言する

## スコープ

- 主対象は**クラウドインフラ**（AWS / GCP / Azure の IaC・構成図）。方法論はベンダ非依存。
- アプリ実装の詳細レビューは対象外。ただし、整合性確認と松竹梅判定のために、アプリ / CI / デプロイ定義は**読む**。必要ならアプリ側修正を「**示唆**」する（アプリの全面レビューはしない）。
- このスキルは**評価レポートのみ**を出力する。TODO 化・実装計画は別スキル「infra-improvement-todo」の責務。
