# 質問シート作成ガイド（Step 6）

Step 6 で生成する「質問シート」の設計指針。**1 枚にまとめて番号付きで一括提示**し、一問ずつ聞かない。ユーザーの往復回数を最小化し、判断材料を一度に揃えるのが目的。

## 提示フォーマット（推奨）

```
以下、確認したい点をまとめました。番号を振ってあるので、まとめてお答えください。
不明・不要な項目は「不明」「該当なし」で構いません。

### ロール関連
1. 「受講希望者」というロールは、社員 / 社外の受講者 / 両方 のどれを想定していますか？
2. ...

### 操作・機能関連
5. ...

### 目的・ビジネス価値関連
9. ...

### 境界・例外関連
12. ...

### 既存機能との境界
15. ...
```

セクションに分けると読みやすい。番号は通し番号で振り、ユーザーが「3, 5, 8 だけ答える」といった応答をしやすくする。

---

## 観点集

各観点はチェックリストではなく **問いの種**。実際の質問文は抽出したストーリー内容に即して具体化する。

### 1. ロールの具体性
- そのロールは単一の主体か？ 複数ロールが混在していないか？
  - 例: 「利用者」が管理者と一般ユーザーの両方を含んでいないか
- 想定するロールに階層はあるか？（例: 一般会員 / プレミアム会員 / 管理者）
- 実運用でそのロールと呼ぶことがあるか？ 命名がユーザーの語彙に合っているか？

### 2. 操作の詳細
- 操作の粒度は適切か？（クリック一発なのか、複数ステップの手続きなのか）
- 前提条件は？（ログイン状態、権限、他機能との依存）
- 操作の結果として何が起きるのか（画面遷移・データ変更・通知）
- 同じ機能を複数の入口から辿れる場合、それは同じストーリーか別ストーリーか

### 3. 目的の明確さ
- なぜその機能が必要なのか、ビジネス価値まで踏み込めているか
- 目的が操作の言い換えになっていないか（例: 「一覧を見るために一覧を見る」）
- 誰にとっての価値か（ユーザー本人 / 別のステークホルダー / 事業）
- KPI に紐づく何かを想定しているか（達成度が測れる目的か）

### 4. 例外・エラー・境界ケース
- 空の状態（データがない、検索結果 0 件、権限なし）はどう扱うか
- エラー時の挙動（通信失敗、バリデーション NG、権限エラー）
- 上限・下限の境界（件数上限、文字数、金額、期間）
- 並行操作・競合（同じデータを同時編集、キャンセル、タイムアウト）
- **本スキルの範囲外**として扱うか、ストーリーに含めるかをユーザーに判断してもらう

### 5. 既存機能との境界
- Step 2 で「除外候補」としてマークした要素は、除外判断で本当に正しいか
- 「既存に見えるが実は差分がある」要素はないか（レイアウトは同じでも挙動が違う等）
- 現行画面比較を行った場合、差分の解釈にズレはないか

### 6. スコープ・優先度
- 今回リリースの対象範囲か、将来対象か
- ストーリー間に依存関係はあるか（A が完了しないと B が意味を持たない、等）
- 一部のストーリーは Epic 分割の再考が必要ではないか

## 質問の作り方 tips

- **選択肢を添える**: 「A / B / 両方 / その他」のような形にすると回答負荷が下がる。
- **YES/NO で答えられる問いは YES/NO で聞く**: オープンクエスチョンばかりだと疲れる。
- **前提が薄い問いは避ける**: 「あなたのプロダクトの成長戦略に照らして…」のような大質問はここでは不要。ストーリー確定に必要な情報だけに絞る。
- **数を絞る**: 質問シート全体で 15〜20 問程度が目安。多すぎると回答されない項目が出る。優先度が低い問いは削る。
- **回答不要マーク**: 「不明・該当なしでも可」を明示すると、ユーザーが立ち止まりにくい。

---

## Step 7 との接続

質問シートに対する回答が来たら、Step 7 で反映する。反映時のポイント:

- **回答で前提が根底から覆るケース**では、部分修正ではなく Step 3 に戻って再抽出する（例: ロール定義が全く違った、除外範囲が大きく変わった等）。
- **軽微な修正**（表現調整・目的の追記）はその場で反映してよい。
- **統合・分割の判断**: 回答を読んだ結果、複数ストーリーを 1 つに統合したり、1 つを分割したりすることがある。件数が 10 件を超えたら Step 4 の絞り込みループへ。
