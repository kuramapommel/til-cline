# ストーリー書式・命名規則リファレンス

## 1. ストーリー本文の書式

```
〈ロール〉として〈操作/機能〉ができる、〈目的〉のため
```

### 各要素の書き方

| 要素 | 説明 | 良い例 | 避ける例 |
|------|------|--------|----------|
| ロール | 単一の主体。組織内の役割・立場で表す | 受講希望者、講師、研修管理者 | ユーザー（曖昧）、Aさん（個人名） |
| 操作/機能 | 動詞を含む具体的な行為 | 研修プログラム一覧を確認する | いい感じにする、色々やる |
| 目的 | ロールにとってのビジネス価値・理由 | 受講可能な研修を把握するため | システムを使うため（トートロジー） |

### チェック観点
- **一文一機能**。「〜と〜ができる」のように 2 つの機能を混ぜない。
- **目的が「機能そのものの言い換え」になっていないか**。例: 「一覧を見るため一覧を見る」は NG。
- **ロールが機能横断で汎用すぎないか**。「ユーザー」は原則使わない。誰が使う機能かを具体化する。

### 抽出例（参考: 研修プログラム一覧画面）
- 受講希望者として「研修プログラム一覧」を確認することができる、受講可能な研修を把握するため
- 受講希望者として各研修プログラムの概略を確認することができる、希望している研修を見つけ出すため
- 受講希望者として特定の研修プログラムをクリックすることでその研修プログラムの詳細ページに遷移することができる、研修プログラムの具体的な内容が希望している内容に適しているのかを判断するため

## 2. ファイル命名規則

### 形式

```
[ID]_[epic-slug]_[story-title-slug].md
```

**ファイル名は英語のみ**を使う。frontmatter の `epic:` フィールドや本文は日本語のままでよい（ファイル名だけを英訳する）。ツールチェーン・URL・シェル・Git ホスティング先での取り回しやすさを優先しての判断。

### 各要素

| 要素 | 生成元 | 例 |
|------|--------|------|
| ID | `scripts/generate_id.js` の出力（URL-safe 10 文字） | `V1StGXR8a9` |
| epic-slug | 確定した Epic 名を英訳し **kebab-case** 化 | `training-list` |
| story-title-slug | ストーリー本文の要点を英訳し **kebab-case** 化 | `view-training-programs` |

### 具体例

```
V1StGXR8a9_training-list_view-training-programs.md
KpL3mN9qZ2_training-list_view-program-summaries.md
Xy7Zw2RtB4_training-list_navigate-to-program-detail.md
```

## 3. 英訳・記号の安全化ルール

ファイル名要素（epic-slug / story-title-slug）は下記の機械的ルールに沿って英語 kebab-case に整形する。

### 3-1. 使用可能文字
- 使えるのは **英小文字 `a-z`、数字 `0-9`、ハイフン `-`** のみ。
- 単語間の区切りは半角ハイフン。アンダースコア `_` は要素間の区切り（`[ID]_[epic]_[title].md`）としてのみ使う。
- **大文字・全角文字・日本語文字は使用禁止**（ID 部分は例外で URL-safe な英数字がそのまま入る）。

### 3-2. 英訳のガイドライン
- 冠詞（a / an / the）は基本省く。トークン数の節約と一貫性のため。
- ストーリータイトルは**動詞の原形で始める**ことを推奨（`view-`, `edit-`, `submit-`, `render-` など。何をする機能かが一目で伝わる）。
- Epic 名も英訳（例: `研修一覧` → `training-list`、`受講申込` → `enrollment`）。
- 英訳が確立していない固有名詞・専門語は、**Hepburn 式ローマ字**または一般的な音訳を採用（例: `絶景アンビエント` → `ambient-scenery` のように意訳、または `zekkei-ambient` のような音訳のいずれか適切な方）。

### 3-3. 長さ制限
- ストーリータイトル部分は**最大 40 文字**（ハイフン含む）に切り詰める。
- 40 文字を超える場合は単語境界で切る（途中の単語が半端に切れないよう調整）。末尾の `-` は削除する。

### 3-4. 禁止文字への保険
- 万一 `/ \ : * ? " < > |` や空白が混じった場合は `-` に置換する。
- 連続するハイフンは 1 つに縮約する（`--` → `-`）。先頭・末尾のハイフンは削除する。

### 3-5. 変換手順（推奨）
1. Epic 名・ストーリータイトルそれぞれを英語に訳す（3-2 のガイドライン）。
2. 英訳結果を小文字化し、単語区切りを `-` に統一する（3-1）。
3. タイトルが 40 文字を超えていれば単語境界で切る（3-3）。
4. 禁止文字・連続ハイフンをクリーンアップ（3-4）。
5. `[ID]_[epic-slug]_[story-title-slug].md` の形で連結する。

### 3-6. frontmatter との関係
- ファイル名は英語だが、**`epic:` フィールドは日本語のままでよい**（そちらは検索・可読性優先）。
- ファイル名の英訳と frontmatter の日本語 Epic 名の対応が分かるよう、両者を対応表として持つと後続工程で便利（本スキルでは求めない）。

## 4. ID 生成

- `node scripts/generate_id.js` を実行して 1 個の ID（10 文字）を取得する。
- 生成器は Node.js 標準の `crypto.randomBytes` を使い、外部 npm 依存はゼロ。
- URL-safe な文字集合（`0-9A-Za-z`、記号なし）を採用しているため、ファイル名・URL・ID として安全に使える。
- 生成された ID は URL・パス・ファイル名で追加のエスケープを必要としない。

### ID 生成器の内部仕様（参考）
- 文字集合: `0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz`（62 文字）
- 長さ: 10 文字
- 空間サイズ: 62^10 ≈ 8.4 × 10^17 通り

同一 ID が偶発的に既存ファイルと衝突した場合は再生成する。開発ライフサイクル終了後にアーカイブされる揮発的な資産であることを前提としており、桁溢れやグローバル一意性は要求しない。

## 5. frontmatter の書き方

出力 Markdown の frontmatter は `assets/story-template.md` の雛形をベースにする。各フィールドの意味と埋め方:

| フィールド | 値 | 例 |
|-----------|-----|------|
| `status` | ライフサイクル状態。本スキルは常に `DRAFT` を出す | `DRAFT` |
| `epic` | Step 5 で確定した Epic 名 | `研修一覧` |
| `tags` | 検索・分類用のタグ配列 | `[研修一覧, 受講希望者, 一覧表示]` |
| `created_at` | 生成日（当日、`YYYY-MM-DD` 形式） | `2026-07-01` |

### tags の埋め方
- 少なくとも Epic 名とロールを含める。
- 機能カテゴリ（例: `一覧表示`、`詳細遷移`、`申込`）を判断できる場合は追加する。判断が難しければ Epic 名とロールだけでもよい。
- タグは日本語で問題ない（frontmatter は YAML なのでカンマ区切りリスト構文 `[a, b, c]` で書く）。

### status の位置付け
`NEW → DRAFT → READY → WIP → DONE → RELEASED / CLOSE` というライフサイクルにおける DRAFT フェーズを表す。受け入れ基準等が空欄で構わないのはこの段階だからで、後続のリファインメントで `READY` に昇格させることを想定している。
