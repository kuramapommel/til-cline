# ユーザストーリー抽出ワークフロー

このスキルは、画面イメージを起点にユーザストーリーを抽出し、対話で確定させたうえで、後続のリファインメントに引き渡せる Markdown ファイル群として出力するためのワークフローです。エージェント（あなた）は画面を読み解く分析者であると同時に、抽出内容の合意形成をリードするファシリテーターでもあります。

## このスキルの立ち位置

- **入力**: 新規画面イメージ（必須） / 現行画面イメージ（任意） / 既存機能宣言（任意） / ここまでに定まっている追加仕様情報（任意。仕様書・Notion・Miro・口頭メモ等）
- **出力**: `docs/userstory/` 配下の Markdown ファイル群（1 ストーリー = 1 ファイル、`status: DRAFT`）。読み取れた UI 情報（レイアウト・カラーパレット）と詳細仕様も記載する。文章化できない UI は画像をストーリーのそばに配置して参照する。
- **範囲**: 「画面 → ストーリー抽出とドラフト作成」まで。**受け入れ基準の具体化はリファインメント（後続工程）の担当**で本スキルの対象外。ただし **UI 情報・詳細仕様など「入力から読み取れる情報」は破棄せず記載する**（`未定義` で機械的に済ませない）。入力からは読み取れず後続で決めるべき情報のみ空欄として残す。

このスキルは「開発ライフサイクル `NEW → DRAFT → READY → WIP → DONE → RELEASED / CLOSE`」における **NEW → DRAFT** のフェーズを担う。ストーリーは開発完了で役割を終える揮発的な資産であることを前提とし、識別子は短いランダム文字列で十分（連番は採番衝突・桁溢れを避けるため使わない）。

## 守るべき基本姿勢

1. **既存機能の除外を勝手に確定しない。** 既存機能宣言と現行画面比較は**除外候補**を挙げるところまで。最終判断は必ずユーザーに確認する。
2. **一度に扱うストーリーは最大 10 件を目安とする。** 認知負荷（Miller の 7±2 / Cowan の 4±1）を踏まえた実用上限。11 件以上あれば絞り込みループに入る。反復漸進的開発にビッグバン的取り込みは適さないと伝え、範囲を絞ってもらう。
3. **質問は 1 枚のシートで一括し、コンソールではなく一時ファイルで行う。** 一問ずつのラリーはユーザーの時間を消費し、コンソールでの入力も負担が大きい。曖昧点・不足情報・境界ケースをまとめた質問シートを `docs/userstory/questions/` に Markdown ファイルとして書き出し、ユーザーにファイルを直接編集して回答してもらう（`persona` / `value-proposition-canvas` と同じ運用）。
4. **入力から読み取れる情報は捨てない。** UI 情報（レイアウト・カラーパレット・コンポーネント）や詳細仕様など、画面イメージや追加仕様から読み取れるものはストーリーに記載する。文章化できない UI は画像をそばに置いて参照する。ただし**創作はしない** — 読み取れないものを推測で埋めず、後続リファインメントや質問シートに委ねる。
5. **回答で前提が覆ったら遠慮なく再抽出。** 部分修正で辻褄合わせをするより、Step 3 に戻って組み直すほうが最終的に速く、精度も高い。その際は必ずユーザーに一言断る。
6. **画像を取得できないときは停止する。** 「たぶんこういう画面だろう」で推測抽出しない。理由を明示して止め、ユーザーに次アクションを委ねる。
7. **出力はコードブロックだけで済ませず、ファイルシステムに書き込む。** 書き込みに失敗した場合のみ、フォールバックとしてコードブロック提示に切り替える。

## 全体フロー（概要）

```
Step 1  入力検証・既存機能の確認・追加仕様情報の収集
Step 2  UI 要素/機能/UI 情報（レイアウト・カラーパレット）の抽出（既存機能を除外候補としてマーク）
Step 3  ユーザストーリー抽出 + Epic 名の自動推測
Step 4  件数チェック（>10 件なら絞り込みループ）
Step 5  Epic 名の確認
Step 6  質問シート生成（docs/userstory/questions/ に一時ファイルとして書き出し）
Step 7  回答反映（前提が覆れば Step 3 へ戻る / 件数超過なら Step 4 へ）
Step 8  Markdown ファイル出力（UI 情報・詳細仕様を記載 / 画像フォールバック配置）
```

各ステップの**詳細手順・分岐条件・停止条件**は `references/workflow.md` に集約している。実行に入る前に必ずそれを読むこと。

## 参照リソース（progressive disclosure）

必要になった時点で読めばよい。冒頭で全て読む必要はない。

- `references/workflow.md` — Step 1〜8 の詳細手順、絞り込みループ、再抽出耐性、停止条件
- `references/story-format.md` — ストーリー書式（〈ロール〉として〈操作/機能〉ができる、〈目的〉のため）、ファイル命名規則、日本語安全化ルール、本文セクションの埋め方、画像ディレクトリの命名
- `references/ui-extraction.md` — UI 情報（レイアウト・カラーパレット・コンポーネント）の抽出観点と、文章化できない場合の画像フォールバック手順
- `references/question-guide.md` — Step 6 の質問シート（一時ファイル）の運用と作成観点集（ロール具体性・目的の明確さ・境界ケース・既存機能との境界）
- `scripts/generate-id.js` — ID 生成スクリプト。**Node.js 標準 `crypto` のみを使用し外部依存ゼロ**（v14 以上を前提）。`node scripts/generate-id.js` で URL-safe 10 文字を出力
- `assets/story-template.md` — 出力 Markdown の雛形（frontmatter + セクション骨格。UI / 画面情報・詳細仕様を含む）。Step 8 でこれを起点に生成する
- `assets/question-sheet-template.md` — Step 6 で書き出す質問シートの雛形

## 出力先とファイル命名

- 出力ディレクトリ: リポジトリルート直下の `docs/userstory/`（存在しなければ作成）。ストーリー Markdown はフラット構成でエピック別サブディレクトリは作らない。
- ファイル名: `[ID]_[epic-slug]_[story-title-slug].md`（**英語 kebab-case のみ**。frontmatter の `epic:` や本文は日本語のままでよい。詳細は `references/story-format.md` を参照）。
- 例: `V1StGXR8a9_training-list_view-training-programs.md`
- **UI 画像フォールバック**: 文章化できない UI 情報があるストーリーには、同名ディレクトリ `docs/userstory/[ID]_[epic-slug]_[story-title-slug]/` を作り、参照画像をそこへ置いてストーリーから相対参照する（`references/ui-extraction.md`）。
- **質問シート（一時ファイル）**: Step 6 の質問シートは `docs/userstory/questions/NNN-<topic-slug>.md` に書き出す。自動削除はしない（`references/question-guide.md`）。

## 前提条件

- `node`（v14 以上）が実行可能であること。未導入の場合は前提未達として理由を説明し停止する。
- 画像・PDF・PPTX などの入力を読み取る手段（Read ツール、MCP、ローカルパスなど）が確保されていること。読めない場合は停止する。
- UI 画像フォールバックを使う場合、参照画像を `docs/userstory/` 配下へコピーできること（ローカルパス等）。貼り付けのみでファイル実体を取得できない場合はコピーできない旨を伝え、ユーザーに配置を依頼するか読み取れた範囲を文章化する。

## 動作言語

対話・出力・生成ファイルの**内容**（frontmatter の値・ストーリー本文・タグ等）はすべて **日本語**。ロール名・機能カテゴリ・タグも日本語で記述してよい。ただし**ファイル名だけは英語 kebab-case** を用いる（`references/story-format.md` の英訳・安全化ルールを適用する）。
