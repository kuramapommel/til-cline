# create-backlog ワークフロー

構造化された仕様を受け取り、Notion のバックログ用データベースに「未実装」状態のアイテムを新規作成する。
バックログアイテムの新規作成までが本スキルの責務であり、ステータス更新・仕様書本文への反映・Issue 作成は行わない。

## ステップ 1 — 入力となる仕様を確認する

**直前のスレッドまたは引数に構造化済みの仕様がある場合**
→「以下の内容でバックログを作成しようとしていると理解しましたが、合っていますか？」と確認し、YES なら次へ進む。

構造化済み仕様として認識する項目：
- 背景 / Context
- ゴール / Goal
- 受け入れ基準 / Acceptance Criteria
- スコープ外 / Out of Scope

**構造化済みの仕様が得られない場合**
→ 以下の項目を一度にまとめてヒアリングする（ユーザーへの往復回数を最小化する）。

| フィールド | ルール |
|---|---|
| 名前（タイトル） | バックログアイテムの名前。一言で「何を実装するか」を表す。 |
| 背景 / Context | なぜこの実装が必要か。課題・背景・きっかけ。 |
| ゴール / Goal | 実装後に達成されるべき状態。実装方法ではなく完了条件を記述する。 |
| 受け入れ基準 / Acceptance Criteria | 「完了」と判断できる具体的な条件。チェックボックス形式が望ましい。 |
| スコープ外 / Out of Scope | やらないこと、変更してはいけないもの。「特になし」も許容する。 |

不明な項目があれば推察して提示し確認する。推察できない場合はそのフィールドについて直接質問する。

## ステップ 2 — 書き込み先データベースを特定する

リポジトリルートの `.project-config.toml` を読み、`[notion]` セクションの `backlog_db_url` を参照する。

```toml
[notion]
backlog_db_url = "https://www.notion.so/..."
```

**値が存在する場合** → その URL を書き込み先として使う。

**ファイルが存在しない、または `backlog_db_url` が未設定の場合**
→ 以下のメッセージをユーザーに伝えて、URL の提示と設定ファイルへの記載を促す。
> 「バックログ用データベースの URL が `.project-config.toml` の `[notion].backlog_db_url` に設定されていません。
>  Notion のバックログ用データベース URL を教えてください。あわせて以下の形式で `.project-config.toml` への追記をお願いします。
>
>  ```toml
>  [notion]
>  backlog_db_url = "https://www.notion.so/..."
>  ```」

## ステップ 3 — Notion MCP の利用可能性を確認する

Notion MCP が利用可能かどうかを確認する。
利用可能かどうかは、セッション開始時に `mcp__notionApi__*` 形式のツールが存在するかで判断できる。

**Notion MCP が使えない場合**は、その旨をユーザーに伝えて作業を止める。
> 「Notion MCP が利用できないため、バックログアイテムを作成できません。
>  Notion MCP の設定については管理者に確認してください。」

## ステップ 4 — 関連仕様書を特定する

バックログ本文の「関連仕様書」欄に記載する既存の確定仕様書ページを特定する。

### 4-1. 関連ページを検索して候補を提示する

仕様の Goal / Context をキーワードに Notion MCP で検索し、関連しそうなページの候補一覧をユーザーに提示する。

```
関連しそうな仕様書が見つかりました。関連仕様書として追加するページを選んでください（複数選択可）：

1. ○○機能設計書 — https://www.notion.so/xxxxx
2. 認証フロー仕様 — https://www.notion.so/yyyyy
（関連なし でも構いません）
```

### 4-2. 追加指定を受け付ける

候補の選択後、「他に追加したい関連仕様書はありますか？URL またはページ名で指定できます」と確認する。
ページ名で指定された場合は Notion MCP で検索して実体を特定し、「このページで合っていますか？」と確認する。

### 4-3. 関連仕様書が存在しない場合

候補が見つからない、またはユーザーが「なし」を選択した場合は、「関連仕様書」見出し配下を空にして次に進む。
関連仕様書がないことを理由に作業を止めない。

## ステップ 5 — 作成内容をユーザーに確認する

アイテムを作成する前に、以下の形式で作成内容をまとめてユーザーに提示し、「この内容でバックログに登録しますか？」と確認する。

---

**📝 バックログ作成内容（確認）**

**名前**: <タイトル>
**ステータス**: 未実装

**背景 / Context**
<内容>

**ゴール / Goal**
<内容>

**受け入れ基準 / Acceptance Criteria**
<内容>

**スコープ外 / Out of Scope**
<内容、または「特になし」>

**関連仕様書**
<ページへのリンク一覧、または「なし」>

---

修正があれば反映して再提示する。確認が取れてから次へ進む。

## ステップ 6 — バックログアイテムを作成する

Notion MCP を使い、特定したデータベースに新規アイテムを作成する。

以下のプロパティとして設定する：
- **名前（タイトル）**: ステップ 1 で確認したタイトル
- **ステータス**: 「未実装」

本文は以下の見出し構成のテンプレートに沿って記載する：

```
## 背景 / Context
<Context の内容>

## ゴール / Goal
<Goal の内容>

## 受け入れ基準 / Acceptance Criteria
<Acceptance Criteria の内容>

## スコープ外 / Out of Scope
<Out of Scope の内容、または「特になし」>

## 関連仕様書
<確定した仕様書ページへのメンションリンク一覧、または空>
```

作成後は、作成したアイテムの Notion MCP からプロパティ情報を取得し、**バックログ ID と URL をユーザーに提示する**。

提示フォーマット：
> バックログに登録しました。ID: PBI-1 / URL: https://www.notion.so/...

バックログ ID がデータベースに存在しない、または取得できなかった場合は以下のメッセージをユーザーに伝える。URL は提示し、作成自体は成功として扱う。
> 「バックログ ID が取得できませんでした。バックログ ID が設定されていない可能性があります。利用しているプロジェクト管理ツールの設定を確認してください。」

## 制約

- **バックログアイテムの新規作成のみを行う**。GitHub Issue 作成・コードベースの探索・ブランチ作成・
  Notion 仕様書本文への書き込み・既存アイテムのステータス更新は、指示がない限り行わない。
- Notion MCP が使えない場合は代替手段を提案せず、MCP の設定を促して止まる。
- 作成前の確認を必ずユーザーから得てから Notion への書き込みを実行する。
- 入力仕様がどのツール・スキルで用意されたかに依存しない。構造化された内容を受け取れればよい。
